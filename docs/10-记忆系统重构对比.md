# Faber 记忆系统重构 —— 原版 vs 新版 详细对比

本文档详细对比 Faber 记忆系统的**原始设计**与**重构后的设计**，帮助理解每个改进的动机、实现方式和工程价值。

---

## 一、数据模型对比

### 1.1 原版 Memory（修改前）

```python
class Memory(BaseModel):
    messages: List[Dict[str, Any]] = Field(default_factory=list)
    # 只有一个扁平的消息列表
```

**特征**：
- 所有消息（system prompt、user、assistant、tool）全部塞进一个 `messages` 列表
- 没有区分消息的"重要性"和"生命周期"
- 压缩时遍历整个列表，一视同仁

**问题**：
- system prompt 和浏览器返回的 5 万字符 HTML 被同等对待
- 跨会话的经验无法独立存储和检索
- 压缩策略粗暴，容易误删重要信息

---

### 1.2 新版 Memory（修改后）

```python
class Memory(BaseModel):
    system_messages: List[Dict[str, Any]] = Field(default_factory=list)
    working_messages: List[Dict[str, Any]] = Field(default_factory=list)
    episodic_notes: List[Dict[str, Any]] = Field(default_factory=list)
```

**三层架构**：

| 层级 | 内容 | 生命周期 | 是否可压缩 |
|------|------|---------|-----------|
| **system_messages** | 系统提示词（角色设定、能力描述、规则约束） | 永久保留 | ❌ 永不压缩 |
| **episodic_notes** | 跨会话经验摘要（"用户经常查天气"、"上次用了搜索工具"） | 跨会话持久 | ❌ 永不压缩 |
| **working_messages** | 当前对话历史（user/assistant/tool 交互） | 当前任务 | ✅ 可压缩 |

**向后兼容**：
- `messages` property 自动合并三层：`system + episodic + working`
- 旧代码访问 `memory.messages` 无需修改，透明返回合并列表
- 数据库旧格式 `{messages: [...]}` 通过 `_from_legacy_messages()` 自动迁移

## 二、持久化机制对比

### 2.1 原版：同步直写

**代码位置**：`app/domain/services/agents/base.py`

```python
async def _add_to_memory(self, messages):
    await self._ensure_memory()
    if self._memory.empty:
        self._memory.add_message({"role": "system", "content": self._system_prompt})
    self._memory.add_messages(messages)
    # 每次添加都同步写数据库！
    async with self._uow:
        await self._uow.session.save_memory(self._session_id, self.name, self._memory)
```

**问题**：
- 每次 LLM 调用前后都要写数据库（一个 ReAct 循环可能调用 10+ 次 LLM）
- 高频对话场景下数据库压力大
- 同步写库阻塞 Agent 执行流程

---

### 2.2 新版：批量异步写入

**新增文件**：`app/infrastructure/memory/db_memory_batch_writer.py`

```python
class DBMemoryBatchWriter(MemoryBatchWriter):
    def __init__(self, uow_factory, batch_size=10, flush_interval=3.0):
        self._queue = asyncio.Queue()
        self._batch_size = batch_size        # 每批最多 10 条
        self._flush_interval = flush_interval  # 每 3 秒 flush 一次
        
    async def enqueue(self, session_id, agent_name, memory):
        await self._queue.put((session_id, agent_name, memory))
        
    async def _flush_loop(self):
        while self._running:
            try:
                await asyncio.wait_for(self._queue.get(), timeout=self._flush_interval)
                if self._queue.qsize() >= self._batch_size - 1:
                    await self._drain_queue()  # 批量写入
            except asyncio.TimeoutError:
                if not self._queue.empty():
                    await self._drain_queue()
```

**关键设计**：
- **去重**：相同 `(session_id, agent_name)` 的写入请求，只保留最新的一份
- **批量**：攒够 10 条或超时 3 秒后，一次性批量写入数据库
- **非阻塞**：`enqueue()` 立即返回，不等待数据库写入
- **容错**：批量写入失败不抛异常，避免打断 Agent 执行

**BaseAgent 中的使用**：

```python
async def _persist_memory(self):
    if self._memory_batch_writer:
        # 批量：入队即可，不等待
        await self._memory_batch_writer.enqueue(self._session_id, self.name, self._memory)
    else:
        # 同步：直接写库（兼容旧模式）
        async with self._uow:
            await self._uow.session.save_memory(...)
```

**main.py 中的生命周期管理**：

```python
# 启动时
await get_memory_batch_writer().start()

# 关闭时
await get_memory_batch_writer().shutdown()  # flush 剩余队列后优雅退出
```

## 三、Token 预算管理对比

### 3.1 原版：无预算概念，固定压缩

**代码位置**：`app/domain/models/memory.py`

```python
def compact(self):
    for message in self.messages:
        if message.get("role") == "tool":
            if message.get("function_name") in ["browser_view", "browser_navigate"]:
                message["content"] = "(removed)"
        if "reasoning_content" in message:
            del message["reasoning_content"]
```

**问题**：
- 不知道当前用了多少 Token
- 无论剩余多少上下文空间，都一视同仁地压缩
- 浏览器结果永远删除，即使上下文还很空
- 没有预防机制，只能事后补救

---

### 3.2 新版：动态预算监控 + 智能压缩

**新增文件**：`app/domain/services/memory/token_counter.py`、`memory_budget.py`

#### 3.2.1 Token 计数器（启发式估算）

```python
class TokenCounter:
    CHINESE_CHAR_TOKEN_RATIO = 1.5   # 中文字符 → Token
    ENGLISH_WORD_TOKEN_RATIO = 1.3   # 英文单词 → Token
    
    @classmethod
    def count_message(cls, message):
        total = 4  # role 固定开销
        content = message.get("content", "")
        total += cls._estimate_text_tokens(content)
        # tool_calls、reasoning_content、tool_call_id 等额外计算
        return total
```

**特点**：
- 不依赖 tiktoken（避免引入外部依赖）
- 中文字符按 1.5x 估算，英文单词按 1.3x 估算
- **高估策略**：宁可多算不可少算，确保不会意外超出模型上下文

#### 3.2.2 预算管理器（三级阈值）

```python
class MemoryBudgetManager:
    COMPACT_SOFT_THRESHOLD = 0.70   # 70%: 记录警告日志
    COMPACT_HARD_THRESHOLD = 0.85   # 85%: 触发自动压缩
    COMPACT_EMERGENCY_THRESHOLD = 0.95  # 95%: 紧急压缩
```

**压缩策略（按价值排序）**：

| 消息类型 | 价值权重 | 压缩行为 |
|----------|---------|---------|
| system prompt | 100 | ❌ 永不压缩 |
| user 消息 | 80 | ❌ 永不压缩 |
| assistant 消息（无 tool_calls） | 70 | ⚠️ 激进模式截断到 200 字 |
| assistant tool_calls | 50 | ❌ 保留（结构完整性） |
| tool 普通结果 | 30 | ✅ 替换为 "(removed)" |
| tool 浏览器结果 | 10 | ✅ 优先删除 |

**智能压缩算法**：

```python
def _compact_by_value(self, memory, aggressive=False):
    # 1. 计算每条消息的价值/Token 比（score = value / tokens）
    scored = []
    for idx, message in enumerate(memory.working_messages):
        tokens = TokenCounter.count_message(message)
        value = self._calculate_message_value(message, idx, total)
        score = value / max(tokens, 1)
        scored.append({"index": idx, "score": score, ...})
    
    # 2. 按 score 升序排序（score 最低的先被压缩）
    scored.sort(key=lambda x: x["score"])
    
    # 3. 依次压缩直到降到目标使用率
    for item in scored:
        if current <= target_usage:
            break
        # 跳过 system 和最近 2 条消息
        if role == "system" or idx >= total - 2:
            continue
        # 执行压缩
        ...
```

**位置因子**：越新的消息价值越高（`recency_factor = 1.0 + index/total * 0.5`），确保最近的对话上下文优先保留。

## 四、压缩策略对比

### 4.1 原版：粗暴替换

```python
# 浏览器结果 → 直接替换为 "(removed)"
message["content"] = "(removed)"

# AI 长回复 → 无处理，保留全文
# 工具结果 → 无处理，保留全文
```

**信息损失**：
- 浏览器返回的 5 万字符 HTML → 变成 `(removed)`，完全不可读
- LLM 后续无法从压缩后的记忆中获取任何信息
- 用户也无法从事件历史中了解 AI 做了什么

---

### 4.2 新版：LLM 智能摘要

**新增文件**：`app/domain/services/memory/memory_summarizer.py`

```python
class MemorySummarizer:
    async def summarize(self, content, context=None):
        prompt = f"""请用一句话总结以下内容的关键信息{context}:
{truncated_content}
要求:
- 保留所有关键数据、结论和事实
- 删除冗余描述和格式
- 长度不超过 200 字
- 直接输出摘要,不要解释"""
        
        result = await self._llm.invoke(messages=[{"role": "user", "content": prompt}])
        return result.get("content", "")
```

**压缩流程**：

```python
async def compact_memory(self):
    # 1. 同步压缩（删除浏览器结果、截断长文本）
    self._memory.compact()
    
    # 2. 异步生成 LLM 摘要（替代粗暴的 "(removed)"）
    if self._summarizer:
        await self._summarizer.batch_summarize(self._memory.working_messages)
    
    # 3. 持久化
    await self._persist_memory()
```

**示例对比**：

| 场景 | 原版 | 新版 |
|------|------|------|
| 浏览器返回 5 万字符 HTML | `(removed)` | `页面包含北京今日天气：晴，25°C，北风3级...` |
| 搜索返回 10 条结果 | `(removed)` | `搜索结果：北京天气晴，上海有雨，广州多云...` |
| Shell 输出 1000 行日志 | `(removed)` | `命令执行成功，输出包含3个警告，无错误...` |

**容错设计**：
- LLM 调用失败时，自动回退到截断策略（保留前 200 字 + `...(truncated)`）
- 不会因为没有 LLM 响应而导致整个压缩流程中断

## 五、长程记忆对比

### 5.1 原版：无跨会话记忆

**问题**：
- 每个 Session 的记忆完全独立
- 用户说"像上次那样处理"，AI 无法获取任何历史上下文
- 相同类型的任务每次都从零开始

---

### 5.2 新版：向量记忆 + 经验检索

**新增文件**：`app/domain/services/memory/vector_memory.py`、`memory_retriever.py`

#### 5.2.1 轻量级向量存储

不使用外部 embedding 服务，而是用**词频向量 + 余弦相似度**：

```python
class VectorMemory:
    def _text_to_vector(self, text):
        # 中文按字符分词，英文按单词分词
        chinese_chars = re.findall(r'[一-鿿]', text)
        english_words = re.findall(r'[a-zA-Z]+', text.lower())
        
        # 去除停用词
        tokens = [c for c in chinese_chars if c not in stop_words]
        tokens += [w for w in english_words if w not in stop_words]
        
        # 归一化词频
        counter = Counter(tokens)
        total = sum(counter.values())
        return {token: count / total for token, count in counter.items()}
    
    @staticmethod
    def _cosine_similarity(v1, v2):
        # 标准余弦相似度计算
        dot_product = sum(v1[w] * v2[w] for w in common)
        return dot_product / (norm1 * norm2)
```

**存储后端**：Redis Hash（每个 Session + Agent 独立存储）

#### 5.2.2 记忆检索器（经验注入）

```python
class MemoryRetriever:
    async def retrieve_for_react(self, query):
        # 用用户查询检索历史经验
        results = await self._react_vm.search(query)
        return results
    
    def format_as_episodic_note(self, result):
        # 格式化为 "[经验] 曾经处理过类似需求..."
        return f"曾经处理过类似需求(相关度{similarity:.0%}): {excerpt}"
```

**注入时机**：`BaseAgent._invoke_llm()` 中，在调用 LLM 之前：

```python
async def _invoke_llm(self, messages, format=None):
    # 1. 添加消息到记忆
    await self._add_to_memory(messages)
    
    # 2. 检索相关历史经验
    if self._memory_retriever and self._memory:
        query = " ".join(msg.get("content", "") for msg in messages if msg.get("role") == "user")
        results = await self._memory_retriever.retrieve_for_react(query)
        
        if results:
            for result in results:
                note = self._memory_retriever.format_as_episodic_note(result)
                self._memory.add_episodic_note(note)  # 注入 episodic_notes
            await self._persist_memory()
    
    # 3. 调用 LLM（此时 LLM 看到的 messages 已包含经验）
    message = await self._llm.invoke(messages=self._memory.get_messages(), ...)
```

**效果**：

```
用户: "帮我查北京天气"

LLM 看到的 messages:
[
  {"role": "system", "content": "你是 Faber..."},           # system_messages
  {"role": "system", "content": "[经验] 曾经处理过类似需求(相关度85%): 用户要求查询天气并生成报告..."},  # episodic_notes
  {"role": "user", "content": "帮我查北京天气"},              # working_messages
]
```

## 七、整体架构对比图

### 7.1 原版记忆系统

```
用户消息 → Agent
              │
              ▼
        Memory.messages (扁平列表)
              │
              ├── system prompt
              ├── user message
              ├── assistant message
              ├── tool result (5万字符 HTML)
              ├── assistant message
              ├── tool result (1万字符搜索)
              └── ... (越来越长)
              │
              ▼
        compact(): 遍历全部，browser 一律替换为 "(removed)"
              │
              ▼
        save_memory(): 同步写 PostgreSQL
              │
              ▼
        LLM.invoke(messages=memory.messages)
```

**问题**：
- ❌ 所有消息混在一个列表，无法区分重要性
- ❌ 每次添加消息同步写库，高频场景性能差
- ❌ 不知道用了多少 Token，被动压缩
- ❌ 压缩粗暴，信息完全丢失
- ❌ 没有跨会话经验积累

---

### 7.2 新版记忆系统

```
用户消息 → Agent
              │
              ▼
        Memory（三层架构）
              │
              ├── system_messages（永不压缩）
              ├── episodic_notes（跨会话经验，永不压缩）
              │     └── 从 VectorMemory 检索注入
              └── working_messages（可压缩）
                    │
                    ├── user message
                    ├── assistant message
                    ├── tool result（长文本）
                    └── ...
              │
              ▼
        _invoke_llm() 阶段:
              1. TokenCounter 计算当前使用量
              2. MemoryBudgetManager 检查预算
              3. 超过阈值 → 按价值排序智能压缩
              4. 压缩后 → MemorySummarizer 生成 LLM 摘要
              │
              ▼
        _persist_memory() 阶段:
              1. MemoryBatchWriter.enqueue() → 异步批量写入
              2. 去重 + 攒批 → 批量 flush 到 PostgreSQL
              │
              ▼
        LLM.invoke(messages=memory.messages)
              ← 包含 system + episodic + working（已压缩+摘要）
```

**改进**：
- ✅ 三层分离，system/episodic 永远安全
- ✅ 批量异步写入，不阻塞 Agent 执行
- ✅ Token 预算实时监控，主动预防溢出
- ✅ 智能压缩按价值排序，优先删除低价值消息
- ✅ LLM 摘要替代粗暴删除，保留关键信息
- ✅ 向量记忆支持跨会话经验检索

## 八、关键设计决策记录

### 8.1 为什么不用 tiktoken？

- 避免新增外部依赖
- 启发式估算对预算管理足够（高估比低估安全）
- 保留接口，未来可无缝替换

### 8.2 为什么向量记忆不用 embedding 服务？

- 避免引入 OpenAI embedding API 或本地模型
- 词频向量 + 余弦相似度对中小规模记忆足够有效
- 实现简单，不依赖外部服务，部署更轻量
- 未来可无缝替换为真实 embedding

### 8.3 为什么 Memory 用 property 兼容旧格式？

- 旧代码大量访问 `memory.messages`
- property 自动合并三层，无需修改所有调用方
- 数据库旧格式通过 `_from_legacy_messages` 自动迁移

### 8.4 为什么 BatchWriter 失败不抛异常？

- 记忆写入失败不应打断 Agent 执行
- 业务逻辑（任务执行）比持久化更重要
- 即使写入失败，内存中的记忆仍然完整，下次有机会再写入

## 九、性能影响预估

| 指标 | 原版 | 新版 | 变化 |
|------|------|------|------|
| 数据库写入次数/轮对话 | 10~20 次 | 1~2 次 | ↓ 90% |
| 单次写入延迟 | ~5ms | ~0ms（异步） | 不阻塞 |
| Token 溢出概率 | 高（被动） | 低（主动预防） | ↓ |
| 上下文有效信息保留率 | ~30%（粗暴删除） | ~70%（智能摘要） | ↑ |
| 跨会话经验复用 | 无 | 有 | 新增 |
