## 1. 核心亮点

### 1.1 多 Agent 协作架构（Planner + ReAct）

项目没有采用单一 Agent 端到端完成所有任务，而是将「规划」与「执行」分离：

- **PlannerAgent**：负责理解用户需求，拆分成带全局 `goal` 的执行计划，并在每步执行后重新评估剩余步骤
- **ReActAgent**：专注于执行单个 step，通过工具循环（Thought → Action → Observation）完成具体动作

文件：`app/domain/services/flows/planner_react.py`

价值：
- 战略层与战术层分离，避免执行 Agent 一边做一边还要想全局
- Planner 可以动态调整路线，相当于任务的「导航仪」
- ReAct 可以专注于当前 step 的工具选择和执行细节

### 1.2 真正的浏览器自动化（远程 CDP + VNC）

浏览器不是跑在 API 进程里的 headless 简化版，而是跑在独立 Docker 沙箱中的完整 Chromium：

- 通过 **CDP（Chrome DevTools Protocol）** 远程连接控制
- 支持 **VNC / Websockify** 远程桌面，开发者可以实时看到浏览器画面
- 页面内容被提取成 Markdown + 编号可交互元素，降低 LLM 理解成本

文件：`app/infrastructure/external/browser/playwright_browser.py`、`faber/sandbox/Dockerfile`、`faber/sandbox/supervisord.conf`

价值：
- 能处理复杂 JS 页面、弹窗、视频、动态加载内容
- 可视化调试方便，VNC 可以直接看到 Agent 在浏览器里做了什么
- 远程 CDP 让浏览器和 API 服务解耦，便于横向扩展

### 1.3 干净可扩展的工具系统

所有工具继承 `BaseTool`，通过 `@tool` 装饰器注册：

```python
class ShellTool(BaseTool):
    @tool(
        name="shell_execute",
        description="在沙箱中执行 Shell 命令",
        parameters={"command": {"type": "string"}},
        required=["command"],
    )
    async def shell_execute(self, command: str) -> ToolResult:
        ...
```

文件：`app/domain/services/tools/base.py`

价值：
- 新增工具只需写方法 + 装饰器
- 自动完成 OpenAI function schema 生成
- 自动过滤 LLM 幻觉产生的多余参数
- 工具调用与事件流天然结合

### 1.4 三层记忆架构与 Token 预算压缩

记忆不是简单的消息列表，而是分为三层：

- `system_messages`：系统提示词，永远保留
- `episodic_notes`：跨会话经验
- `working_messages`：当前对话历史

`MemoryBudgetManager` 根据价值加权压缩：

- 浏览器结果最长最占 token，优先压缩
- 系统提示词和最近 2 条消息不压缩
- 用 LLM 生成语义摘要替代粗暴删除

文件：`app/domain/models/memory.py`、`app/domain/services/memory/memory_budget.py`、`app/domain/services/memory/memory_summarizer.py`

价值：
- 在长任务中控制上下文长度，避免超出模型窗口
- 保护核心上下文不被误删
- 压缩后的记忆仍有语义可用性

### 1.5 多层防跑偏机制

项目在长任务稳定性上投入了大量设计：

- Plan 动态更新（不修改全局 goal）
- 全局计划上下文注入 ReAct
- 周期性反思检查点
- Step 验收条件（success_criteria）
- 单 step 迭代预算
- 任务级超时
- 提前完成检测

文件：`docs/19-长任务防跑偏机制.md`

价值：
- 不是单一机制，而是多层防护网
- 从计划、执行、记忆、提示词多个维度降低跑偏概率
- 是项目比较有特色的部分

### 1.6 外部生态集成（MCP + A2A）

- **MCP（Model Context Protocol）**：对接外部 MCP 服务，如高德地图、Jina
- **A2A（Agent-to-Agent）**：调用远程 Agent

文件：`app/domain/services/tools/mcp.py`、`app/domain/services/tools/a2a.py`

价值：
- 项目不是封闭的，能接入开放协议的外部能力
- 未来可以方便地扩展更多专业工具

### 1.7 事件驱动 + SSE 实时推送

整个执行过程基于异步事件流：

```python
PlanEvent / StepEvent / ToolEvent / MessageEvent / DoneEvent / ErrorEvent
```

所有事件持久化到数据库，同时通过 SSE 实时推送给前端。

文件：`app/domain/models/event.py`、`app/interfaces/endpoints/session_routes.py`

价值：
- 前端可以实时看到 Agent 的思考和执行过程
- 事件持久化支持事后复盘、调试、审计
- 异步架构适合长任务和流式响应

## 2. 主要技术难点

### 2.1 长任务跑偏

这是项目最核心的难点。虽然做了多层防护，但本质仍依赖 LLM 自律：

- ReAct 虽然有全局上下文，但执行粒度仍是 step 级
- 反思检查点依赖 LLM 自省，LLM 自身幻觉时反思也可能无效
- 没有独立的 critic agent 对当前路径做客观评判
- 提前完成检测需要额外 LLM 调用，有成本和误判风险

文件：`docs/19-长任务防跑偏机制.md` 第 10 章

### 2.2 浏览器自动化稳定性

浏览器自动化有很多工程细节容易出问题：

- 只扫描**视口内**可交互元素，页面外元素必须先滚动才能发现
- 多 tab 场景处理比较脆弱，当前实现取 `pages[-1]`
- 坐标点击依赖视口大小，窗口变化后坐标会失效
- 复杂 SPA 页面的动态加载、弹窗、iframe、Shadow DOM 都是潜在坑
- 页面内容转 Markdown 会丢失链接和图像，不保证完整

文件：`app/infrastructure/external/browser/playwright_browser.py`、`app/infrastructure/external/browser/playwright_browser_fun.py`

### 2.3 上下文窗口管理

长任务中记忆会迅速膨胀：

- 浏览器结果动辄几千 token
- 搜索摘要、shell 输出、文件内容都会占上下文
- 压缩策略虽然智能，但仍可能误删关键信息
- 默认 8000 token 预算对现代长上下文模型来说偏低

文件：`app/domain/services/memory/memory_budget.py`

### 2.4 多 Agent 状态协调

Planner 和 ReAct 有各自记忆、各自调用 LLM，协调复杂：

- ReAct 执行完 step 后，结果要正确回流给 Planner 更新 plan
- plan 更新时不能破坏已完成的 steps
- 会话恢复、用户中途打断、等待用户输入等状态转换容易出错
- 两个 Agent 的记忆需要分别压缩和持久化

文件：`app/domain/services/flows/planner_react.py`

### 2.5 Docker 沙箱的运维成本

沙箱是项目的重要基础设施，但也带来运维负担：

- 需要本机安装 Docker
- 沙箱镜像 2GB+，首次构建时间长
- 沙箱容器需要健康检查、自动创建、销毁、重启
- 容器内同时运行 Chromium、VNC、supervisor、FastAPI 等多个进程
- 网络、文件系统、权限隔离需要仔细处理

文件：`faber/sandbox/Dockerfile`、`faber/sandbox/supervisord.conf`、`app/infrastructure/external/sandbox/docker_sandbox.py`

### 2.6 LLM 输出的不可控性

LLM 不是确定性系统，输出格式和质量都有波动：

- Planner 必须输出合法 JSON plan，但 LLM 可能漏字段或格式错误
- ReAct 必须输出符合格式的 step 结果
- 工具参数可能有幻觉，虽然 `_filter_parameters` 做了过滤，但值本身仍可能不对
- 反思检查点的回答质量不稳定

文件：`app/domain/services/agents/base.py`、`app/domain/services/agents/planner.py`、`app/domain/services/agents/react.py`

### 2.7 异步事件流的调试难度

整个 flow 是 async generator，事件流贯穿多个 Agent、工具、数据库：

- 某个环节出错时，定位困难
- 事件顺序、并发、回滚逻辑复杂
- 前端 SSE 断连重连也要考虑
- 工具事件中的 `tool_content`（如浏览器截图）需要特殊处理

文件：`app/domain/services/agent_task_runner.py`、`app/application/services/agent_service.py`

### 2.8 本地部署门槛

项目要完整跑起来，依赖较多：

- Python 3.12+
- Docker + 2GB+ 沙箱镜像
- PostgreSQL、Redis
- 可选的 COS / OSS / LLM API Key

对本地开发环境要求不低。

## 3. 亮点与难点的对应关系

| 亮点 | 解决的难点 | 仍未完全解决的问题 |
|---|---|---|
| Planner + ReAct 双 Agent | 长任务无全局规划 | Planner 更新仍依赖 LLM，没有客观评判 |
| 浏览器自动化 | 需要访问真实网页 | 视口限制、多 tab、动态页面稳定性 |
| 工具装饰器 | 工具扩展性差 | LLM 工具参数幻觉 |
| 记忆预算压缩 | 上下文窗口超限 | 可能误删关键信息 |
| 多层防跑偏 | 任务跑偏 | 仍依赖 LLM 自省 |
| MCP / A2A | 外部能力扩展 | 外部服务不可用时的降级 |
| 事件驱动 + SSE | 前端无法实时观察 | 事件流复杂，调试困难 |

## 4. 综合评价

| 维度 | 评分 | 说明 |
|---|---|---|
| 架构设计 | ⭐⭐⭐⭐☆ | 分层清晰，多 Agent + 事件驱动 |
| 工程完整度 | ⭐⭐⭐⭐☆ | 工具、记忆、沙箱、浏览器、外部集成都有 |
| 长任务稳定性 | ⭐⭐⭐☆☆ | 机制很多，但仍依赖 LLM 自律 |
| 可扩展性 | ⭐⭐⭐⭐☆ | MCP/A2A + 工具装饰器，扩展方便 |
| 本地部署门槛 | ⭐⭐☆☆☆ | 需要 Docker + Python 3.12 + 沙箱镜像 |
| 代码质量 | ⭐⭐⭐☆☆ | 整体清晰，但有些地方有重复代码和未清理的遗留文件 |

### 适合场景

- 需要 Agent 自主执行多步骤任务
- 需要浏览器自动化访问真实网页
- 需要沙箱隔离执行代码和命令
- 愿意投入运维成本换取 Agent 能力

### 不适合场景

- 追求极简部署
- 任务非常简单，不需要多 Agent 协作
- 对 LLM 输出稳定性要求极高的场景
