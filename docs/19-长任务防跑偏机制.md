## 1. 概述：什么是「跑偏」

在 Agent 执行长任务时，"跑偏"指 Agent 的实际行为逐渐偏离用户原始目标，表现为：

| 跑偏类型 | 典型表现 |
|---|---|
| 目标漂移 | 解决了某个中间问题，却忘了最终要交付什么 |
| 钻牛角尖 | 在某个失败步骤上反复重试，消耗大量迭代 |
| 上下文腐化 | 记忆压缩后丢失关键信息，导致后续决策基于错误假设 |
| 发散浏览/搜索 | 被无关链接或搜索结果带离主线 |
| 过度执行 | 任务已完成仍继续执行剩余步骤 |
| 提前终止 | 尚未完成目标就以为任务结束 |

Faber 的防跑偏不是单一机制，而是**多层防护网**。

## 2. 架构层：Planner + ReAct 双 Agent 分离

### 2.1 设计思想

将「规划」与「执行」分离：
- **PlannerAgent**：负责把用户请求拆成 plan，并在每步执行后重新评估计划
- **ReActAgent**：负责执行单个 step，通过工具循环完成任务

文件：`app/domain/services/flows/planner_react.py:34-101`

```python
class PlannerReActFlow(BaseFlow):
    def __init__(self, ...):
        tools = [FileTool, ShellTool, BrowserTool, SearchTool, MessageTool, mcp_tool, a2a_tool]
        self.planner = PlannerAgent(..., tools=tools)
        self.react = ReActAgent(..., tools=tools)
```

### 2.2 Flow 状态机

```
IDLE → PLANNING → EXECUTING → UPDATING → EXECUTING → ... → SUMMARIZING → COMPLETED
```

文件：`app/domain/services/flows/base.py:9-16`

每个状态有明确职责：
- `PLANNING`：创建 plan
- `EXECUTING`：取出下一个 step 让 ReAct 执行
- `UPDATING`：根据 step 结果更新 plan
- `SUMMARIZING`：汇总所有 step 结果生成最终回复

### 2.3 防跑偏价值

- Planner 定期重新评估全局路径，相当于「战略层」校准
- ReAct 专注于当前 step，相当于「战术层」执行
- 战略与战术分离，避免执行 Agent 一边做一边还要想全局

## 3. 计划层：Plan 的结构化跟踪与动态更新

### 3.1 Plan 模型

文件：`app/domain/models/plan.py:32-51`

```python
class Plan(BaseModel):
    id: str
    title: str          # 任务标题
    goal: str           # 任务目标（全局目标）
    language: str       # 工作语言
    steps: List[Step]   # 步骤列表
    message: str        # 初始回复
    status: ExecutionStatus

    def get_next_step(self) -> Optional[Step]:
        return next((step for step in self.steps if not step.done), None)
```

```python
class Step(BaseModel):
    id: str
    description: str
    success_criteria: Optional[str] = None  # 步骤验收标准
    status: ExecutionStatus
    result: Optional[str]
    error: Optional[str]
    success: bool
    attachments: List[str]

    @property
    def done(self) -> bool:
        return self.status in [COMPLETED, FAILED]
```

关键设计：
- `goal` 字段保存全局目标，贯穿整个任务
- 每个 step 有独立状态（PENDING/RUNNING/COMPLETED/FAILED）
- `success_criteria` 描述该步骤的验收标准，什么情况下算成功
- `get_next_step()` 始终取第一个未完成的 step，保证执行顺序

- `goal` 在 plan 更新时被保护，不易漂移
- 每步完成后重新规划，失败时可以调整后续路径
- 已完成的 steps 不会被修改，保证历史一致性
- `success_criteria` 让每个 step 有明确完成标准，避免模糊执行
- 不必要的步骤可以被删除，避免过度执行

## 4. 执行层：ReAct 循环的约束与反思

### 4.1 ReAct 执行循环

文件：`app/domain/services/agents/base.py:273-345`

```python
async def invoke(self, query, format=None, plan_context=None):
    message = await self._invoke_llm([{"role": "user", "content": query}], format)

    # ReAct执行使用单步迭代预算，Planner调用使用全局预算
    iteration_limit = (
        min(self._agent_config.max_iterations_per_step, self._agent_config.max_iterations)
        if plan_context
        else self._agent_config.max_iterations
    )

    for loop_index in range(1, iteration_limit + 1):
        if not message or not message.get("tool_calls"):
            break

        tool_messages = []
        for tool_call in message["tool_calls"]:
            # 执行工具，yield ToolEvent
            result = await self._invoke_tool(tool, function_name, function_args)
            tool_messages.append({"role": "tool", ...})

        # 周期性反思
        if plan_context and loop_index % reflection_interval == 0:
            reflection = self._build_reflection_message(plan_context)
            tool_messages.insert(0, {"role": "user", "content": reflection})

        message = await self._invoke_llm(tool_messages)
    else:
        yield ErrorEvent(error=f"Agent迭代超过最大迭代次数: {iteration_limit}, 任务处理失败")
```

### 4.2 硬性约束

#### 最大迭代次数

- 防止 Agent 整体无限工具循环

```python
@staticmethod
def detect_early_completion(text: Optional[str]) -> bool:
    if not text:
        return False
    markers = [
        "[EARLY_COMPLETE]",
        "已提前完成",
        "任务已提前达成",
        "early complete",
```

```python
当前已完成步骤: {completed_step.description}
该步骤结果摘要: {completed_step.result or "无"}
剩余待执行步骤: {...}
```

在 `EXECUTING` 状态，step 完成后：

```python
if (
    self._agent_config.enable_early_completion
    and step.result
    and self.react.detect_early_completion(step.result)

    remaining_steps = [s for s in self.plan.steps if not s.done and s.id != step.id]
```

### 4.7 防跑偏价值

- 执行 Agent 每轮都看到全局目标，减少局部优化
- 定期强制 LLM 自省，及时拉回偏离主线
- 单 step 有迭代预算，防止钻牛角尖
- 任务有总超时，防止无限 hang 住
- 目标提前达成时可跳过剩余步骤，避免过度执行

```python
def _compact_by_value(self, memory: Memory, aggressive: bool = False):
    messages = memory.messages
    scored_messages = []
    for idx, message in enumerate(messages):
        tokens = TokenCounter.count_message(message)
        value = self._calculate_message_value(message, idx, len(messages))
        score = value / max(tokens, 1)
        scored_messages.append({"index": idx, "message": message, "score": score})

    scored_messages.sort(key=lambda x: x["score"])

    target_usage = self._budget * 0.60 if aggressive else self._budget * 0.70

    for item in scored_messages:
```

### 5.4 消息价值计算

文件：`app/domain/services/memory/memory_budget.py:185-214`

```python
def _calculate_message_value(self, message, index, total):
    role = message.get("role", "")
    base_value = self.MESSAGE_VALUE_WEIGHTS.get(role, 50)

    if role == "tool":
        func_name = message.get("function_name", "")
        if func_name in ["browser_view", "browser_navigate"]:
            base_value = self.MESSAGE_VALUE_WEIGHTS["tool_browser"]

    recency_factor = 1.0 + (index / max(total, 1)) * 0.5
    return base_value * recency_factor
```

### 5.5 LLM 驱动的记忆摘要

文件：`app/domain/services/memory/memory_summarizer.py:9-122`

- 在同步压缩后，异步为被压缩的消息生成 LLM 摘要
- 用摘要替代粗暴的 `(removed)`，保留更多语义信息
- 失败时回退到截断

### 5.6 防跑偏价值

- 系统提示词和最近 2 条消息永远保留，保证核心上下文不丢
- 浏览器结果（最长最占 token）优先压缩，防止上下文被页面内容淹没
- 价值加权 + 位置加权，既保重要信息，也保新鲜信息
- LLM 摘要让压缩后的记忆仍可用

### 6.3 防跑偏价值

- 经验质量取决于历史任务的成功率和摘要质量

```
- 仅重新规划后续**未完成**的步骤，不要更改已完成的步骤
- 你可以删除、添加或者修改计划步骤，但不要改变计划目标 (goal)
- 为每个步骤添加 success_criteria，描述该步骤完成的具体可验证标准
```

### 7.4 语言一致性约束

文件：`app/domain/services/prompts/system.py:14-20`

```
- 默认工作语言：**中文 (Chinese)**
- 所有的思考过程（Thinking）和回复必须使用工作语言
- 工具调用（Tool calls）中的自然语言参数必须使用工作语言
```

### 7.5 防跑偏价值

- 反复强调「亲自执行」、「交付结果而非计划」
- Planner 被约束不能修改 goal
- `success_criteria` 让每个 step 有明确标准
- 语言一致性防止多语言切换导致的理解偏差

## 9. 各层协同工作示例

假设用户请求：「搜索北京今天天气，打开新浪首页截图，把结果写成 /tmp/result.txt」

1. **PlannerAgent** 创建 plan：
   - goal: "搜索北京天气并截图，将结果写入文件"
   - steps:
     - 搜索北京天气（success_criteria: 获得北京当天温度、天气状况）
     - 打开新浪首页截图（success_criteria: 成功截取新浪首页图片）
     - 写入文件（success_criteria: /tmp/result.txt 存在且包含天气信息）

2. **Flow** 进入 EXECUTING，取出 step 1

3. **ReActAgent.execute_step()** 构建 plan_context：
   - goal、title、已完成（空）、待执行（截图、写文件）
   - success_criteria 传入 prompt

4. **ReActAgent** 执行 step 1，调用 `search_web`

5. **BaseAgent.invoke()** 循环：
   - loop_index=1: 调用 search_web → 拿到结果 → 不触发反思（假设 interval=5）
   - 生成 step 结果，返回 MessageEvent

6. **Flow** 进入 UPDATING，Planner 更新 plan：
   - 已完成的搜索保留
   - 剩余步骤：[截图, 写文件]

7. **Flow** 回到 EXECUTING，取出 step 2

8. ReAct 执行 step 2，plan_context 更新：
   - 已完成：[搜索天气]
   - 待执行：[写文件]

9. 如果 step 2 中浏览器打开了很多无关页面，loop_index 达到 5 时：
   - 触发反思检查点
   - LLM 看到提示：「我当前的操作是否仍然对齐原始目标？」
   - 可能决定关闭无关页面，回到新浪首页

10. 如果 step 2 的结果意外包含了全部所需信息，且 `enable_early_completion=True`：
    - `detect_early_completion()` 识别到提前完成标记
    - `_check_goal_satisfied()` 判断全局目标已达成
    - Flow 跳过 step 3，直接进入 SUMMARIZING

11. 所有 steps 完成后，Flow 进入 SUMMARIZING，ReAct 汇总结果

12. 生成 DoneEvent，任务结束

## 10. 当前仍存在的局限与后续方向

### 10.1 已实现的方向

| 方向 | 状态 | 实现位置 |
|---|---|---|
| Step 验收条件 | ✅ 已完成 | `app/domain/models/plan.py`, `app/domain/services/prompts/planner.py`, `app/domain/services/prompts/react.py` |
| 任务级超时 + 单步迭代预算 | ✅ 已完成 | `app/domain/models/app_config.py`, `app/domain/services/flows/planner_react.py`, `app/domain/services/agents/base.py` |
| 提前完成检测 | ✅ 已完成 | `app/domain/models/app_config.py`, `app/domain/services/prompts/react.py`, `app/domain/services/agents/react.py`, `app/domain/services/flows/planner_react.py` |

### 10.2 仍待实现的方向

| 局限/方向 | 说明 | 优先级 |
|---|---|---|
| 没有独立 critic Agent | 缺少专门模块评判当前路径是否正确，完全依赖 LLM 自省 | 中 |
| 没有 wall-clock timeout 的细分 | 当前只有任务级超时，没有单个工具调用的超时 | 低 |
| ReAct 仍以 step 为粒度 | 即使看到全局，也不会在 step 内部自动调整策略 | 中 |
| 反思依赖 LLM 自省 | LLM 自身幻觉时，反思也可能无效 | 中 |
| Plan adherence 量化评分 | 当前只有定性反思，没有定量分数 | 低 |
| Vector Memory 语义能力有限 | TF-IDF 无法捕捉深层语义相似 | 低 |

### 10.3 后续增强建议

1. **添加轻量 critic agent**
   - 每轮评估 step 结果是否匹配 `success_criteria`
   - 发现偏离时触发 replan 或 user confirm

2. **工具级超时**
   - 为 Docker 沙箱调用、浏览器操作等设置单独超时

3. **增强 Vector Memory**
   - 接入 embedding 模型，提升语义检索质量

4. **Plan adherence 量化评分**
   - 让 LLM 给当前行为与 plan 的对齐度打分
   - 分数低于阈值时触发告警或 replan

## 11. 配置参考

在 `config.yaml` 的 `agent_config` 段可配置：

```yaml
agent_config:
  max_iterations: 100              # Agent 全局最大迭代次数
  max_iterations_per_step: 20      # ReAct 单 step 最大迭代次数
  max_retries: 3                   # LLM/工具 失败重试次数
  max_search_results: 10           # 搜索返回最大条数
  reflection_interval: 5           # 每 N 轮工具循环反思一次，0 关闭
  task_timeout_seconds: 600        # 任务级超时时间（秒）
  enable_early_completion: true    # 是否启用提前完成检测
```

调整建议：
- **短任务（1-3 轮工具调用）**：`reflection_interval=0` 或 `20`，`max_iterations_per_step=10`
- **复杂研究/浏览器任务**：`reflection_interval=3-5`，`max_iterations_per_step=20-30`
- **调试模式**：`reflection_interval=1`，可观察每次反思输出
- **严格时限场景**：`task_timeout_seconds=300`
