# Faber API —— 关键设计模式与工程经验

本章总结 Faber 项目中反复出现的设计模式，以及从代码中提炼的工程经验。这些知识不仅适用于本项目，也可迁移到其他 AI Agent 系统或复杂后端系统的设计中。

---

## 1. Clean Architecture（整洁架构）

### 1.1 模式定义

将系统分为四层，依赖关系**由内向外**：
- **领域层（Domain）**：业务规则、实体、价值对象
- **应用层（Application）**：用例编排、事务管理
- **接口层（Interfaces）**：HTTP/WebSocket 暴露、序列化
- **基础设施层（Infrastructure）**：数据库、外部 API、消息队列

### 1.2 Faber 中的体现

```
domain/external/llm.py          # Protocol（只定义接口）
infrastructure/external/llm/openai_llm.py  # 实现（依赖领域层）
```

领域层的 `LLM` 是一个 Protocol，不依赖 OpenAI SDK。基础设施层的 `OpenAILLM` 导入并实现了 `LLM`。

### 1.3 工程价值

| 价值 | 说明 |
|------|------|
| 可测试性 | 领域层可用内存中的 FakeLLM、FakeSandbox 测试，无需真实外部服务 |
| 可替换性 | 今天用 DeepSeek，明天换 Claude，只需新增一个 LLM 实现类 |
| 可维护性 | 业务逻辑集中在一处，不会散落到 SQL 查询或 HTTP 调用中 |

## 2. 依赖注入（DI）

### 2.1 模式定义

不直接在类内部创建依赖，而是通过构造函数传入。由外部"组装器"负责创建和连接对象。

### 2.2 Faber 中的体现

```python
# service_dependencies.py 作为"组装器"
def get_agent_service() -> AgentService:
    llm = OpenAILLM(app_config.llm_config)
    file_storage = OssFileStorage(...)
    return AgentService(
        uow_factory=get_uow,
        llm=llm,
        sandbox_cls=DockerSandbox,
        task_cls=RedisStreamTask,
        ...
    )
```

### 2.3 为什么选择手动 DI？

Faber 没有使用 `dependency-injector` 或 `injector` 等框架，原因：
1. **显式优于隐式**：代码即文档，一眼能看到依赖关系
2. **生命周期复杂**：某些依赖（如 `app_config`）需要运行时读取文件，框架 DI 的生命周期管理反而复杂
3. **Python 的灵活性**：函数是一等公民，工厂函数足够表达大多数 DI 场景

## 3. 协议驱动设计（Protocol-Driven Design）

### 3.1 模式定义

使用 Python 的 `typing.Protocol`（PEP 544）定义接口，而非抽象基类（ABC）。

### 3.2 Faber 中的体现

```python
class LLM(Protocol):
    async def invoke(self, messages, tools=None, ...) -> Dict[str, Any]: ...

class Sandbox(Protocol):
    async def exec_command(self, session_id, exec_dir, command) -> ToolResult: ...
```

### 3.3 Protocol vs ABC

| 特性 | ABC | Protocol |
|------|-----|----------|
| 继承关系 | 必须显式继承 | 结构化子类型（duck typing）|
| 运行时检查 | `isinstance()` 可用 | 需要 `typing.runtime_checkable` |
| 灵活性 | 低（强制继承链） | 高（任何实现方法即可）|
| IDE 支持 | 好 | 好 |

**Faber 的选择**：Protocol 更灵活，尤其适合"不希望引入继承关系"的场景。

## 4. 仓库模式 + UoW（Unit of Work）

### 4.1 模式定义

- **Repository**：封装对数据集合的访问，像操作内存中的集合一样操作数据库
- **Unit of Work**：维护一个变更列表，一次性提交所有变更，确保事务原子性

### 4.2 Faber 中的体现

```python
class IUnitOfWork(ABC):
    file: FileRepository
    session: SessionRepository
    
    async def __aenter__(self): ...
    async def __aexit__(self, exc_type, ...): ...

class DBUnitOfWork(IUnitOfWork):
    async def __aenter__(self):
        self.db_session = self.session_factory()
        self.file = DBFileRepository(self.db_session)
        self.session = DBSessionRepository(self.db_session)
        return self
    
    async def __aexit__(self, exc_type, ...):
        if exc_type:
            await self.rollback()
        else:
            await self.commit()
        await self.db_session.close()
```

### 4.3 使用方式

```python
async with uow:
    await uow.session.save(session)
    await uow.file.save(file)
    # 退出时自动 commit 或 rollback
```

### 4.4 防御性设计：SSE 断连处理

```python
async def __aexit__(self, exc_type, exc_val, exc_tb):
    try:
        if exc_type:
            await self.rollback()
        else:
            await self.commit()
    except asyncio.CancelledError:
        logger.warning("UoW 提交/回滚操作被取消")
    except Exception as e:
        logger.warning(f"UoW 提交/回滚操作失败: {e}")
    finally:
        try:
            await self.db_session.close()
        except asyncio.CancelledError:
            logger.warning("UoW 关闭数据库会话被取消")
```

**经验**：在长连接（SSE/WebSocket）场景中，`CancelledError` 是常态而非异常。必须确保资源释放不受 cancel scope 影响。

## 5. 状态机（State Machine）

### 5.1 模式定义

用有限状态和转移规则描述复杂流程，避免嵌套 if-else。

### 5.2 Faber 中的体现

`PlannerReActFlow` 是一个经典的状态机：

```python
class FlowStatus(str, Enum):
    IDLE = "idle"
    PLANNING = "planning"
    EXECUTING = "executing"
    UPDATING = "updating"
    SUMMARIZING = "summarizing"
    COMPLETED = "completed"
```

状态转移：
```
IDLE → PLANNING → EXECUTING → UPDATING → EXECUTING → ... → SUMMARIZING → COMPLETED → IDLE
```

### 5.3 工程价值

1. **可预测性**：每个状态的行为明确，不会进入"未知状态"
2. **可观察性**：可以实时显示"AI 正在规划中..."
3. **可恢复性**：从数据库恢复 Session 后，根据状态可以重建 Flow

## 6. 生产者-消费者模式

### 6.1 模式定义

生产者生成数据放入队列，消费者从队列取出数据处理。两者解耦，可独立扩展。

### 6.2 Faber 中的体现

```
AgentTaskRunner（生产者）        AgentService.chat()（消费者）
      │                                │
      ▼                                ▼
task.output_stream.put(event)    task.output_stream.get(event)
      │                                │
      └────────── Redis Stream ────────┘
```

### 6.3 工程价值

- **解耦**：生产者不关心消费者的速度和数量
- **背压**：Redis Stream 天然缓冲，消费者跟不上时不会丢数据
- **可扩展**：未来可将生产者和消费者部署在不同服务上

---

## 7. 事件溯源（Event Sourcing）思想

### 7.1 模式定义

不保存对象的当前状态，而是保存导致状态变更的所有事件。当前状态可通过重放事件推导。

### 7.2 Faber 中的体现

Session 的 `events` 字段是 JSONB 数组：

```json
[
  {"type": "message", "role": "user", "message": "..."},
  {"type": "plan", "plan": {"steps": [...]}},
  {"type": "step", "step": {"id": "1", "status": "started"}},
  {"type": "tool", "tool_name": "search", "status": "calling"},
  {"type": "tool", "tool_name": "search", "status": "called"},
  {"type": "step", "step": {"id": "1", "status": "completed"}},
  ...
]
```

### 7.3 工程价值

- **审计**：完整记录 AI 的每一步操作
- **调试**：可以重放事件还原问题现场
- **恢复**：服务重启后，从事件重建会话状态

## 8. 装饰器注册模式

### 8.1 模式定义

通过 Python 装饰器将元数据绑定到函数/方法上，运行时通过反射收集。

### 8.2 Faber 中的体现

```python
def tool(name, description, parameters, required):
    def decorator(func):
        func._tool_schema = {...}
        func._tool_name = name
        return func
    return decorator

class MyTool(BaseTool):
    @tool(name="do_something", description="...", parameters={...}, required=[...])
    async def do_something(self, ...): ...
```

`BaseTool.get_tools()` 通过 `inspect.getmembers()` 反射收集所有带 `_tool_schema` 的方法。

### 8.3 工程价值

- **声明式**：开发者只需关注业务逻辑，注册自动完成
- **零配置**：新增工具无需修改注册表、配置文件
- **自文档**：装饰器参数即 OpenAI Function Schema

---

## 9. 全局注册表（Registry）

### 9.1 模式定义

在类级别维护一个字典，存储活跃对象的引用，便于快速查找。

### 9.2 Faber 中的体现

```python
class RedisStreamTask(Task):
    _task_registry: Dict[str, "RedisStreamTask"] = {}
    
    def __init__(self, ...):
        RedisStreamTask._task_registry[self._id] = self
    
    @classmethod
    def get(cls, task_id: str) -> Optional["Task"]:
        return RedisStreamTask._task_registry.get(task_id)
```

### 9.3 工程价值

- **O(1) 查找**：通过 ID 直接获取对象，无需查询数据库或 Redis
- **生命周期管理**：任务取消时从注册表移除，防止内存泄漏

---

## 10. 单例模式（Singleton）

### 10.1 模式定义

确保一个类只有一个实例，并提供全局访问点。

### 10.2 Faber 中的体现

```python
@lru_cache()
def get_postgres() -> Postgres:
    return Postgres()

@lru_cache()
def get_redis() -> RedisClient:
    return RedisClient()
```

使用 `@lru_cache()`（无参数）实现单例：第一次调用创建实例，后续调用返回缓存的实例。

### 10.3 与显式单例的区别

| 方式 | 代码 | 特点 |
|------|------|------|
| 显式单例 | `class Singleton: _instance = None; def __new__(...)` | 侵入类定义 |
| lru_cache | `@lru_cache() def get_xxx()` | 非侵入，可缓存清除 |

Faber 选择 `lru_cache` 是因为可以方便地清除缓存（在 `shutdown()` 中调用 `cache_clear()`）。

## 11. 工厂模式（Factory）

### 11.1 模式定义

用工厂方法或工厂类创建对象，而不是直接用构造函数。

### 11.2 Faber 中的体现

```python
class DockerSandbox(Sandbox):
    @classmethod
    async def create(cls) -> Self:
        # 复杂创建逻辑：Docker SDK、IP 解析...
        return DockerSandbox(ip=ip, container_name=name)
    
    @classmethod
    async def get(cls, id: str) -> Optional[Self]:
        # 复杂查找逻辑：检查容器状态...
        return DockerSandbox(ip=ip, container_name=id)
```

### 11.3 工程价值

- **封装复杂性**：创建沙箱涉及 Docker SDK、网络配置、错误处理，工厂方法隐藏了这些细节
- **多态创建**：`AgentService` 通过 `sandbox_cls=DockerSandbox` 注入，未来可换为 `KataSandbox`

## 12. 策略模式（Strategy）

### 12.1 模式定义

定义一系列算法，将它们封装起来，并且使它们可以互相替换。

### 12.2 Faber 中的体现

```python
class StatusService:
    def __init__(self, checkers: List[HealthChecker]):
        self._checkers = checkers
    
    async def check_all(self) -> List[HealthStatus]:
        return [await checker.check() for checker in self._checkers]
```

`PostgresHealthChecker` 和 `RedisHealthChecker` 实现相同的 `HealthChecker` 协议，StatusService 不关心具体实现。

---

## 13. 透明代理模式（Transparent Proxy）

### 13.1 模式定义

在两个通信方之间转发数据，本身不解析或处理协议内容。

### 13.2 Faber 中的体现

WebSocket VNC 代理：
```python
async def forward_to_sandbox():
    while True:
        data = await websocket.receive_bytes()
        await sandbox_ws.send(data)

async def forward_from_sandbox():
    while True:
        data = await sandbox_ws.recv()
        await websocket.send_bytes(data)
```

Faber 后端不解析 VNC 协议，只是双向转发字节流。

### 13.3 工程价值

- **无状态**：不维护 VNC 会话状态，可水平扩展
- **低延迟**：无需解析和重组数据包
- **安全**：在转发前可做鉴权（如检查 session 是否存在）

## 14. 重试与熔断

### 14.1 模式定义

- **重试**：操作失败时自动重试，提高成功率
- **熔断**：连续失败时快速失败，避免级联故障

### 14.2 Faber 中的体现

```python
# BaseAgent._invoke_llm() 中的重试
for _ in range(self._agent_config.max_retries):
    try:
        return await self._llm.invoke(...)
    except Exception as e:
        await asyncio.sleep(self._retry_interval)
        continue
raise RuntimeError("调用语言模型失败")
```

Faber 实现了重试，但没有显式熔断。这是因为：
- LLM 服务（DeepSeek）的故障通常是瞬时的（网络抖动）
- 每个请求独立，不会因一个请求失败影响其他请求
- 如果 LLM 完全不可用，错误会快速返回给前端

**未来增强**：可引入 `tenacity` 库的指数退避 + 熔断器。

## 15. 防御性编程（Defensive Programming）

### 15.1 参数过滤

```python
@classmethod
def _filter_parameters(cls, method: Callable, kwargs: Dict) -> Dict:
    """LLM 可能产生幻觉参数，只保留方法签名中存在的参数"""
    filtered = {}
    sign = inspect.signature(method)
    for key, value in kwargs.items():
        if key in sign.parameters:
            filtered[key] = value
    return filtered
```

### 15.2 空值处理

```python
if message.get("role") == "assistant":
    if not message.get("content") and not message.get("tool_calls"):
        # DeepSeek 思考模型可能返回空 content
        logger.warning("LLM回复了空内容，执行重试")
```

### 15.3 连接池保护

```python
# DBUnitOfWork.__aexit__ 中的多重异常捕获
try:
    await self.commit()
except asyncio.CancelledError:
    logger.warning("UoW 提交被取消")
except Exception as e:
    logger.warning(f"UoW 提交失败: {e}")
finally:
    try:
        await self.db_session.close()
    except asyncio.CancelledError:
        ...
```

## 16. 提示词工程（Prompt Engineering）最佳实践

### 16.1 结构化输出

所有需要结构化的场景使用 `json_object` 格式：
```python
_format: str = "json_object"
```

### 16.2 TypeScript 接口定义

用 TypeScript 描述期望的 JSON Schema（LLM 对类型系统理解较好）：
```
interface Response {
  success: boolean;
  attachments: string[];
  result: string;
}
```

### 16.3 示例输出

提供具体示例，降低格式错误率：
```json
{
  "success": true,
  "result": "任务已完成...",
  "attachments": ["/home/ubuntu/report.md"]
}
```

### 16.4 约束强化

关键约束多次重复：
- "是你来执行这个任务，而不是用户"
- "严禁使用列表格式"
- "必须亲自执行任务，而不是指导用户去执行"

### 16.5 工作语言一致性

强制所有思考和输出使用同一语言：
```
- 所有的思考过程和回复必须使用工作语言
- 工具调用中的自然语言参数必须使用工作语言
```

## 17. 可扩展性设计

### 17.1 新增 LLM 提供商

1. 创建 `infrastructure/external/llm/claude_llm.py`
2. 实现 `LLM` Protocol
3. 在 `service_dependencies.py` 中根据配置选择实现类

无需修改领域层代码。

### 17.2 新增工具

1. 在 `domain/services/tools/` 创建新工具类
2. 继承 `BaseTool`，用 `@tool` 装饰方法
3. 在 `PlannerReActFlow.__init__()` 中添加到工具列表

无需修改 Agent 逻辑。

### 17.3 新增 MCP 服务

在 `config.yaml` 中添加：
```yaml
mcp_config:
  mcpServers:
    my-new-service:
      transport: streamable_http
      url: https://api.example.com/mcp
```

无需修改代码，重启服务即可生效。

## 18. 调试与运维经验

### 18.1 日志级别策略

| 级别 | 用途 |
|------|------|
| DEBUG | 消息队列读写、记忆操作、工具参数 |
| INFO | 服务初始化、状态转移、任务起停 |
| WARNING | 重试、空回复、客户端断连 |
| ERROR | 工具调用失败、LLM 调用失败、沙箱创建失败 |

### 18.2 关键日志字段

所有日志都包含 `[session_id]` 和 `[task_id]`，便于追踪：
```
INFO: 会话[abc-123]已启动
INFO: 任务[def-456]开始执行
INFO: Planner&ReAct流开始执行步骤 step-1: 搜索天气信息
```

### 18.3 健康检查

`/api/status` 检查：
- PostgreSQL：`SELECT 1`
- Redis：`PING`
- FastAPI：自身响应时间

应增加：
- LLM 服务商可用性
- 沙箱镜像是否存在
- 磁盘空间（防止日志/容器占满）

## 19. 已知局限与未来优化

### 19.1 当前局限

1. **单点任务执行**：一个 Task 在一个 asyncio Task 中执行，无法利用多核 CPU
2. **无持久化任务队列**：Redis Stream 在内存中，重启后丢失未消费的消息
3. **沙箱无预热**：每次创建会话需要等待容器启动（10-30 秒）
4. **无熔断机制**：LLM 服务完全不可用时，请求会堆积

### 19.2 未来优化方向

1. **任务队列持久化**：使用 RabbitMQ / Kafka，支持服务重启后恢复
2. **沙箱池化**：预启动一批沙箱容器，用户创建会话时直接分配
3. **并行步骤执行**：无依赖关系的步骤可并行执行（如同时搜索多个关键词）
4. **人机协作增强**：支持用户中途修改计划、批准/拒绝特定工具调用
5. **向量记忆**：将历史会话向量化，支持跨会话记忆检索

---

## 20. 总结

Faber 的架构设计体现了以下核心思想：

1. **分层解耦**：Clean Architecture 让业务逻辑独立于技术细节
2. **协议驱动**：Protocol 让系统可插拔、可测试
3. **事件优先**：所有交互以事件为媒介，天然支持流式、异步、可观测
4. **容错设计**：重试、防御性异常处理、参数过滤，应对 LLM 的不确定性
5. **扩展友好**：新增工具、LLM、MCP 服务都只需最小改动

这些设计模式和工程经验，是 Faber 能够处理复杂 AI 任务、保持稳定运行的基石。
