# Faber API —— 领域层（Domain）：模型与事件系统

领域层是整个系统的**核心与灵魂**。它包含：
- **领域模型**：表达业务概念的数据结构
- **业务规则**：如"计划必须有步骤"、"已完成的步骤不能重新执行"
- **Agent 智能体逻辑**：Planner、ReAct、Flow 状态机
- **工具系统**：Agent 能调用的所有工具
- **外部服务协议**：定义系统需要哪些外部能力（但不实现它们）

**核心原则**：领域层**不依赖任何其他层**，甚至不依赖 FastAPI、SQLAlchemy、Redis、Docker SDK。它只使用 Python 标准库和 Pydantic。

---

## 1. 领域模型总览

```
Session（会话）
  ├── events: List[Event]      # 所有历史事件
  ├── files: List[File]        # 会话关联的文件
  ├── memories: Dict[str, Memory]  # Agent 记忆（planner + react）
  ├── plan: Plan（从 events 中提取）
  └── status: SessionStatus

Event（事件）— 前后端通信的通用语言
  ├── MessageEvent   # 用户/AI 消息
  ├── PlanEvent      # 计划生成/更新
  ├── StepEvent      # 步骤状态变化
  ├── ToolEvent      # 工具调用
  ├── TitleEvent     # 会话标题
  ├── WaitEvent      # 等待用户输入
  ├── DoneEvent      # 任务结束
  └── ErrorEvent     # 错误

Plan（计划）
  ├── steps: List[Step]
  ├── goal: str
  ├── title: str
  └── status: ExecutionStatus

Step（步骤）
  ├── description: str
  ├── status: ExecutionStatus
  └── result: Optional[str]
```

## 2. Session（会话模型）

```python
class SessionStatus(str, Enum):
    PENDING = "pending"      # 空闲，等待第一条消息
    RUNNING = "running"      # Agent 正在执行任务
    WAITING = "waiting"      # 等待用户输入（如 message_ask_user）
    COMPLETED = "completed"  # 任务已完成

class Session(BaseModel):
    id: str = Field(default_factory=lambda: str(uuid.uuid4()))
    sandbox_id: Optional[str] = None
    task_id: Optional[str] = None
    title: str = ""
    unread_message_count: int = 0
    latest_message: str = ""
    latest_message_at: Optional[datetime] = None
    events: List[Event] = Field(default_factory=list)
    files: List[File] = Field(default_factory=list)
    memories: Dict[str, Memory] = Field(default_factory=dict)
    status: SessionStatus = SessionStatus.PENDING
    updated_at: datetime = Field(default_factory=datetime.now)
    created_at: datetime = Field(default_factory=datetime.now)
```

### 2.1 设计要点

**为什么 events 存在 Session 中，而不是单独的数据库表？**

因为事件是**严格顺序的、只追加的、同会话内查询的**。使用 PostgreSQL 的 JSONB 数组存储：
- 避免了复杂的多表 JOIN
- 原子性更新（一条 UPDATE 语句追加事件）
- 单个会话的事件量通常在百级以内，JSONB 性能足够

**sessions 表的关键字段**：

```sql
CREATE TABLE sessions (
    id VARCHAR(255) PRIMARY KEY,
    events JSONB NOT NULL DEFAULT '[]',
    memories JSONB NOT NULL DEFAULT '{}',
    files JSONB NOT NULL DEFAULT '[]',
    status VARCHAR(255) NOT NULL DEFAULT ''
);
```

### 2.2 获取最新计划

```python
def get_latest_plan(self) -> Optional[Plan]:
    for event in reversed(self.events):
        if isinstance(event, PlanEvent):
            return event.plan
    return None
```

倒序遍历事件列表，找到最后一个 PlanEvent。这体现了**事件溯源（Event Sourcing）**的思想：当前状态是从历史事件中推导出来的。

## 3. Event（事件系统）

事件是 Faber 的**通用语言**。前后端通过事件通信，Agent 内部也通过事件传递信息。

### 3.1 事件类型详解

#### MessageEvent —— 消息事件

```python
class MessageEvent(BaseEvent):
    type: Literal["message"] = "message"
    role: Literal["user", "assistant"] = "assistant"
    message: str = ""
    attachments: List[File] = Field(default_factory=list)
```

- `role="user"`：用户发送的消息
- `role="assistant"`：AI 发送的消息

#### PlanEvent —— 计划事件

```python
class PlanEvent(BaseEvent):
    type: Literal["plan"] = "plan"
    plan: Plan
    status: PlanEventStatus = PlanEventStatus.CREATED
```

- `status=CREATED`：首次生成计划
- `status=UPDATED`：根据执行结果更新计划
- `status=COMPLETED`：所有步骤执行完毕

#### StepEvent —— 步骤事件

```python
class StepEvent(BaseEvent):
    type: Literal["step"] = "step"
    step: Step
    status: StepEventStatus = StepEventStatus.STARTED
```

前端通过 StepEvent 实时显示"AI 正在执行第 3 步：搜索相关信息"。

#### ToolEvent —— 工具事件（最复杂！）

```python
class ToolEvent(BaseEvent):
    type: Literal["tool"] = "tool"
    tool_call_id: str
    tool_name: str           # 工具集名字，如 "browser", "shell"
    function_name: str       # 具体工具名字，如 "browser_navigate"
    function_args: Dict[str, Any]
    function_result: Optional[ToolResult] = None
    tool_content: Optional[ToolContent] = None  # 前端展示的富内容
    status: ToolEventStatus = ToolEventStatus.CALLING
```

**为什么需要 tool_content？**

`function_result` 是原始的工具返回结果（给 LLM 看的），而 `tool_content` 是前端展示的**富媒体内容**：
- Browser 工具：`tool_content` 包含截图 URL
- Search 工具：`tool_content` 包含搜索结果列表
- Shell 工具：`tool_content` 包含控制台输出

**ToolContent Union**：

```python
ToolContent = Union[
    BrowserToolContent,   # {screenshot: str}
    SearchToolContent,    # {results: List[SearchResultItem]}
    ShellToolContent,     # {console: Any}
    FileToolContent,      # {content: str}
    MCPToolContent,       # {result: Any}
    A2AToolContent,       # {a2a_result: Any}
]
```

#### WaitEvent —— 等待事件

```python
class WaitEvent(BaseEvent):
    type: Literal["wait"] = "wait"
```

当 Agent 调用 `message_ask_user` 工具时，会触发 WaitEvent。前端收到后：
- 显示输入框让用户回复
- 将会话状态改为 WAITING
- 用户回复后，再次调用 `/chat` 接口

### 3.2 Event Union 与类型安全

```python
Event = Annotated[
    Union[
        PlanEvent, TitleEvent, StepEvent,
        MessageEvent, ToolEvent, WaitEvent,
        ErrorEvent, DoneEvent,
    ],
    Field(discriminator="type"),
]
```

使用 Pydantic 的 **discriminated union**：根据 `type` 字段自动反序列化为正确的子类。

```python
# 自动识别为 ToolEvent
event = TypeAdapter(Event).validate_json('{"type": "tool", "tool_name": "browser", ...}')
assert isinstance(event, ToolEvent)
```

## 4. Plan & Step（计划与步骤）

### 4.1 Plan

```python
class Plan(BaseModel):
    id: str = Field(default_factory=lambda: str(uuid.uuid4()))
    title: str = ""
    goal: str = ""
    language: str = ""       # 工作语言（如 "zh", "en"）
    steps: List[Step] = Field(default_factory=list)
    message: str = ""        # AI 生成计划时的回复消息
    status: ExecutionStatus = ExecutionStatus.PENDING
    error: Optional[str] = None
    
    @property
    def done(self) -> bool:
        return self.status in [ExecutionStatus.COMPLETED, ExecutionStatus.FAILED]
    
    def get_next_step(self) -> Optional[Step]:
        return next((step for step in self.steps if not step.done), None)
```

### 4.2 Step

```python
class Step(BaseModel):
    id: str = Field(default_factory=lambda: str(uuid.uuid4()))
    description: str = ""    # 步骤描述（给 ReActAgent 看的）
    status: ExecutionStatus = ExecutionStatus.PENDING
    result: Optional[str] = None   # 执行结果
    error: Optional[str] = None    # 错误信息
    success: bool = False
    attachments: List[str] = Field(default_factory=list)  # 生成的文件路径
    
    @property
    def done(self) -> bool:
        return self.status in [ExecutionStatus.COMPLETED, ExecutionStatus.FAILED]
```

### 4.3 ExecutionStatus 状态机

```
PENDING → RUNNING → COMPLETED
                ↘ FAILED
```

### 4.4 计划的动态更新

PlannerAgent 不是一次性生成完美计划，而是**根据执行结果动态调整**：

1. 初始生成计划（3-5 步）
2. ReActAgent 执行第 1 步
3. 根据执行结果，PlannerAgent 更新剩余步骤（可能删除、修改、新增）
4. ReActAgent 执行下一步
5. ...

这种**渐进式规划**更符合复杂任务的实际执行过程，因为 LLM 无法一次性预见所有细节。

## 5. Memory（记忆模型）

每个 Agent（Planner 和 ReAct）都有自己的 Memory，本质是 OpenAI 格式的消息列表。

```python
class Memory(BaseModel):
    messages: List[Dict[str, Any]] = Field(default_factory=list)
    
    def add_message(self, message: Dict[str, Any]) -> None:
        self.messages.append(message)
    
    def add_messages(self, messages: List[Dict[str, Any]]) -> None:
        self.messages.extend(messages)
    
    def get_last_message(self) -> Optional[Dict[str, Any]]:
        return self.messages[-1] if self.messages else None
    
    def roll_back(self) -> None:
        """删除最后一条消息"""
        self.messages = self.messages[:-1]
    
    def compact(self) -> None:
        """压缩记忆，减少 Token 消耗"""
        for message in self.messages:
            if message.get("role") == "tool":
                # 删除浏览器页面源码（太占 Token）
                if message.get("function_name") in ["browser_view", "browser_navigate"]:
                    message["content"] = "(removed)"
            # 删除思考内容（DeepSeek 的 reasoning_content）
            if "reasoning_content" in message:
                del message["reasoning_content"]
```

### 5.1 记忆的持久化

Memory 存储在 Session 的 `memories` 字段中：

```json
{
  "planner": {
    "messages": [
      {"role": "system", "content": "..."},
      {"role": "user", "content": "帮我查天气"},
      {"role": "assistant", "content": "{\"steps\": [...]}"}
    ]
  },
  "react": {
    "messages": [...]
  }
}
```

### 5.2 为什么需要两个 Memory？

Planner 和 ReAct 是两个独立的 Agent：
- **Planner 的记忆**：包含用户原始需求、历史计划、步骤执行结果（用于更新计划）
- **ReAct 的记忆**：包含当前步骤的详细执行过程、工具调用历史（用于执行具体任务）

分开存储避免了上下文混淆，也让各自的 prompt 更聚焦。

### 5.3 记忆压缩（Compact）

浏览器工具返回的页面内容可能非常长（数万个字符）。如果全部保留在记忆中：
- Token 消耗爆炸
- LLM 上下文窗口溢出
- 无关信息干扰后续推理

`compact()` 策略：
- 浏览器访问结果 → 替换为 "(removed)"
- 思考过程（reasoning_content）→ 删除

**时机**：ReActAgent 每执行完一个 Step 后调用 `compact()`。

## 6. Message（消息模型）

```python
class Message(BaseModel):
    message: str = ""
    attachments: List[str] = Field(default_factory=list)
```

这是 Agent 内部流转的简化消息模型。与 `MessageEvent` 不同：
- `MessageEvent` 是**领域事件**（带 ID、时间戳、类型）
- `Message` 是**业务对象**（仅内容和附件路径）

当用户发送消息时：
1. 接口层创建 `ChatRequest`
2. 应用层创建 `MessageEvent` 并持久化
3. 领域层的 Flow 接收 `Message`（从 MessageEvent 提取）

## 7. File（文件模型）

```python
class File(BaseModel):
    id: str = Field(default_factory=lambda: str(uuid.uuid4()))
    filename: str = ""
    filepath: Optional[str] = None    # 沙箱中的路径
    key: Optional[str] = None         # OSS/COS 中的对象键
    url: Optional[str] = None         # 访问 URL
    size: int = 0
    mime_type: str = ""
```

### 7.1 文件流转

用户上传文件 → 存储到 OSS/COS → 生成 `File` 记录 → 用户发消息时附带 File ID → AgentService 将文件**同步到沙箱** → AI 在沙箱中操作文件 → AI 生成新文件 → 新文件**同步回 OSS/COS** → 前端通过 URL 访问

## 8. ToolResult（工具结果）

```python
class ToolResult(BaseModel, Generic[T]):
    success: bool = True
    message: Optional[str] = ""
    data: Optional[T] = None
    
    @classmethod
    def from_sandbox(cls, code: int, msg: str, data: Optional[T], **kwargs):
        return cls(
            success=True if code < 300 else False,
            message=msg,
            data=data,
        )
```

使用泛型 `ToolResult[SearchResults]` 可以精确表达"搜索工具返回的是 SearchResults 类型"。

## 10. 外部服务协议（External Protocols）

领域层定义了系统需要的外部能力，但不关心具体实现。

### 10.1 LLM 协议

```python
class LLM(Protocol):
    async def invoke(
        self,
        messages: List[Dict[str, Any]],
        tools: List[Dict[str, Any]] = None,
        response_format: Dict[str, Any] = None,
        tool_choice: str = None,
    ) -> Dict[str, Any]: ...
    
    @property
    def model_name(self) -> str: ...
    @property
    def temperature(self) -> float: ...
    @property
    def max_tokens(self) -> int: ...
```

**为什么用 Protocol 不用 ABC？**

Python 的 `Protocol`（PEP 544）是**结构化子类型**：只要类实现了协议中的方法，就自动被视为该协议的子类，无需显式继承。这比 ABC 更灵活，尤其是在不想引入继承关系时。

### 10.2 Sandbox 协议

这是最大的协议，定义了沙箱必须提供的所有能力：

```python
class Sandbox(Protocol):
    # Shell 操作
    async def exec_command(self, session_id, exec_dir, command) -> ToolResult: ...
    async def read_shell_output(self, session_id, console=False) -> ToolResult: ...
    
    # 文件操作
    async def read_file(self, filepath, ...) -> ToolResult: ...
    async def write_file(self, filepath, content, ...) -> ToolResult: ...
    async def replace_in_file(self, filepath, old_str, new_str) -> ToolResult: ...
    async def upload_file(self, file_data, filepath, filename) -> ToolResult: ...
    async def download_file(self, filepath) -> BinaryIO: ...
    
    # 沙箱生命周期
    async def ensure_sandbox(self) -> None: ...
    async def destroy(self) -> bool: ...
    async def get_browser(self) -> Browser: ...
    
    # 类方法：工厂
    @classmethod
    async def create(cls) -> Self: ...
    @classmethod
    async def get(cls, id: str) -> Optional[Self]: ...
    
    # 属性
    @property
    def id(self) -> str: ...
    @property
    def vnc_url(self) -> str: ...
    @property
    def cdp_url(self) -> str: ...
```

### 10.3 Browser 协议

```python
class Browser(Protocol):
    async def view_page(self) -> ToolResult: ...
    async def navigate(self, url: str) -> ToolResult: ...
    async def click(self, index=None, coordinate_x=None, coordinate_y=None) -> ToolResult: ...
    async def input(self, text, press_enter, index=None, ...) -> ToolResult: ...
    async def screenshot(self, full_page=None) -> bytes: ...
    async def console_exec(self, javascript: str) -> ToolResult: ...
```

### 10.4 Task 协议

```python
class Task(Protocol):
```

## 11. 仓库接口（Repository Interfaces）

### 11.1 UnitOfWork 协议

```python
class IUnitOfWork(ABC):
    file: FileRepository
    session: SessionRepository
    
    @abstractmethod
    async def commit(self): ...
    @abstractmethod
    async def rollback(self): ...
    @abstractmethod
    async def __aenter__(self): ...
    @abstractmethod
    async def __aexit__(self, exc_type, exc_val, exc_tb): ...
```

### 11.2 SessionRepository

```python
class SessionRepository(ABC):
    @abstractmethod
    async def save(self, session: Session) -> None: ...
    @abstractmethod
    async def get_all(self) -> List[Session]: ...
    @abstractmethod
    async def get_by_id(self, session_id: str) -> Optional[Session]: ...
    @abstractmethod
    async def delete_by_id(self, session_id: str) -> None: ...
    @abstractmethod
    async def update_title(self, session_id: str, title: str) -> None: ...
    @abstractmethod
    async def add_event(self, session_id: str, event: BaseEvent) -> None: ...
    @abstractmethod
    async def save_memory(self, session_id: str, agent_name: str, memory: Memory) -> None: ...
    @abstractmethod
    async def get_memory(self, session_id: str, agent_name: str) -> Memory: ...
```

**设计模式**：Repository 屏蔽了数据访问细节。领域层只说"保存会话"，不关心是存 PostgreSQL、MongoDB 还是内存。
