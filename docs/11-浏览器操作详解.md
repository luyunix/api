## 1. 整体架构

浏览器不是跑在 API 进程里的，而是跑在一个独立的 Docker 沙箱容器中：

```
API 进程 (macOS/本地)
  └── DockerSandbox
        └── 启动 faber-sandbox 容器
              ├── Xvfb 虚拟显示器 :1
              ├── Chromium 浏览器（CDP 端口 8222）
              ├── socat 把 8222 映射到 0.0.0.0:9222
              ├── x11vnc (5900) + websockify (5901) 提供远程桌面
              └── 沙箱内部 FastAPI (8080)
```

API 通过 `cdp_url = http://{sandbox_ip}:9222` 用 Playwright 的 `connect_over_cdp()` 远程操控浏览器。

## 2. 抽象层：Browser Protocol

代码里不直接依赖 Playwright，而是先定义了一个 `Protocol`：

```python
# app/domain/external/browser.py:6
class Browser(Protocol):
    async def view_page(self) -> ToolResult: ...
    async def navigate(self, url: str) -> ToolResult: ...
    async def click(self, index=None, coordinate_x=None, coordinate_y=None) -> ToolResult: ...
    async def input(self, text, press_enter, index=None, x=None, y=None) -> ToolResult: ...
    async def screenshot(self, full_page=None) -> bytes: ...
    async def console_exec(self, javascript: str) -> ToolResult: ...
    # ...
```

这样 Agent、Tool、TaskRunner 都只依赖接口，以后换 Selenium/Puppeteer 也不影响上层。

### 3.1 连接与初始化

特点：
- 重试 5 次，指数退避（最多 10s）
- 只识别 `about:blank` / `chrome://newtab/` 为空白页
- 多 tab 时取 `pages[-1]` 作为当前页

### 3.2 页面内容提取

```python
# playwright_browser.py:69-95
async def _extract_content(self) -> str:
    visible_content = await self.page.evaluate(GET_VISIBLE_CONTENT_FUNC)
    markdown_content = markdownify(visible_content)
    max_content_length = min(len(markdown_content), 50000)
    if self.llm:
        # 如果有 LLM，会让 LLM 再整理一次 markdown
        response = await self.llm.invoke(...)
        return response.get("content", "")
    return markdown_content[:max_content_length]
```

- 通过 JS 脚本抓取**可视区域**内所有元素的 `outerHTML`
- `markdownify` 转成 Markdown
- 截断到 50K 字符
- 可选让 LLM 二次提炼

### 3.3 可交互元素标记

这是给 LLM「看」页面的核心机制：

```python
# playwright_browser_fun.py:50-196
GET_INTERACTIVE_ELEMENTS_FUNC = """() => {
    const elements = document.querySelectorAll(
        'button, a, input, textarea, select, [role="button"], [tabindex]:not([tabindex="-1"])'
    );
    ...
    element.setAttribute('data-faber-id', `faber-element-${validElementIndex}`);
    interactiveElements.push({
        index: validElementIndex,
        tag: tagName,
        text: text,
        selector: selector
    });
    ...
}"""
```

流程：
1. 查询所有可交互元素
2. 过滤掉不在视口内、宽高为 0、`display:none` 等元素
3. 给每个元素打上 `data-faber-id="faber-element-0/1/2..."`
4. 返回 `{index, tag, text, selector}` 列表

Agent 之后就可以用 `browser_click(index=2)` 这样的整数索引来操作页面，而不是脆弱的 XPath/CSS。

### 3.4 点击与输入

```python
# playwright_browser.py:132-181
async def click(self, index=None, coordinate_x=None, coordinate_y=None):
    if x is not None and y is not None:
        await self.page.mouse.click(x, y)
    elif index is not None:
        element = await self._get_element_by_id(index)
        # 如果不可见，先 scrollIntoView
        if not is_visible:
            await self.page.evaluate("element.scrollIntoView(...)", element)
        await element.click(timeout=5000)
```

```python
# playwright_browser.py:332-...
async def input(self, text, press_enter, index=None, x=None, y=None):
    if index is not None:
        element = await self._get_element_by_id(index)
        await element.fill("")      # 先清空
        await element.type(text)    # 再输入
        if press_enter:
            await self.page.keyboard.press("Enter")
```

支持两种定位方式：
- `index`：通过 `data-faber-id` 精确命中
- `coordinate_x/y`：直接模拟鼠标坐标点击/输入

## 4. 工具层：`BrowserTool`

`BrowserTool` 把 `Browser` 接口包装成 LLM 可调用的 Function Calling 工具：

```python
# app/domain/services/tools/browser.py:8
class BrowserTool(BaseTool):
    name: str = "browser"

    @tool(name="browser_navigate", ...)
    async def browser_navigate(self, url: str) -> ToolResult:
        return await self.browser.navigate(url)

    @tool(name="browser_click", ...)
    async def browser_click(self, index=None, x=None, y=None) -> ToolResult:
        return await self.browser.click(index, x, y)

    @tool(name="browser_input", ...)
    async def browser_input(self, text, press_enter, index=None, x=None, y=None) -> ToolResult:
        return await self.browser.input(text, press_enter, index, x, y)
    # ... 还有 scroll、screenshot、console_exec、console_view 等 12 个
```

`@tool` 装饰器（在 `base.py` 里）会给方法挂上 OpenAI 格式的 schema，包括 `name/description/parameters/required`。Planner 把这些 schema 发给 LLM，LLM 决定调用哪个。

## 5. Agent 如何调用浏览器

在 `PlannerReActFlow` 初始化时，浏览器工具被注册进工具链：

```python
# app/domain/services/flows/planner_react.py:62-71
tools = [
    FileTool(sandbox=sandbox),
    ShellTool(sandbox=sandbox),
    BrowserTool(browser=browser),   # 浏览器工具
    SearchTool(search_engine=search_engine),
    MessageTool(),
    mcp_tool,
    a2a_tool,
]
```

`BaseAgent.invoke()` 是核心循环：

```python
# app/domain/services/agents/base.py:273-344
async def invoke(self, ...):
    while True:
        response = await self.llm.invoke(messages, tools=self.tool_schemas)
        if response.tool_calls:
            # 每次只取第一个 tool_call
            tool_call = response.tool_calls[0]
            yield ToolEvent(status=CALLING, ...)
            result = await self._invoke_tool(tool_call)
            yield ToolEvent(status=CALLED, function_result=result, ...)
```

也就是说：
1. LLM 看到用户请求后，可能决定调用 `browser_navigate`
2. Agent 执行工具，得到 `ToolResult`
3. 结果再放回 messages，进入下一轮思考/行动

## 8. 当前实现的优势与局限

### 8.1 优势
- **真浏览器**：不是无头简化版，是完整 Chromium，能跑复杂 JS、处理弹窗、视频等
- **可远程观察**：VNC/Websockify 让开发者能实时看到浏览器画面
- **LLM 友好**：把页面转成 Markdown + 编号元素，LLM 更容易理解
- **解耦**：Protocol + Playwright 实现，换底层驱动成本低

### 8.2 局限
1. **只能看到视口内元素**：`GET_INTERACTIVE_ELEMENTS_FUNC` 只扫描当前可视区域，页外元素需要先 `scroll`
2. **单页假设**：`_ensure_page()` 取 `pages[-1]`，多 tab 场景容易混乱
3. **无持久状态**：`browser_restart` 会清掉所有 Playwright 资源，不能恢复会话
4. **截图开销**：每次浏览器操作后都上传一张 PNG，高频操作下 OSS/COS 费用和延迟需要考虑
5. **坐标脆弱**：`coordinate_x/y` 是视口相对坐标，窗口大小变化会失效
6. **内存压缩偏见**：`memory_budget.py` 里浏览器结果价值分较低（`tool_browser=10`），长会话中会被优先压缩
