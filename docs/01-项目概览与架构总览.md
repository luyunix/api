# Faber API —— 项目概览与架构总览

## 1. 项目定位

Faber 是一个**可完全私有部署的通用 AI Agent（智能体）系统后端**。它的核心使命是：让用户通过与 AI 对话，驱动 AI 在**安全的隔离环境（Docker 沙箱）**中完成复杂任务。

这些任务包括但不限于：
- 信息收集、事实核查、深度研究
- 网页浏览、数据抓取
- 编写和运行代码（Python、Node.js、Shell 等）
- 文件读写、编辑、分析
- 调用外部 MCP（Model Context Protocol）服务（如高德地图、Jina AI）
- 调用其他远程 A2A（Agent-to-Agent）Agent

**产品形态**：前端（不在本项目中）通过 SSE（Server-Sent Events）实时观看 AI 的"思考过程"——包括计划生成、工具调用、步骤执行、最终结果——并可通过 WebSocket VNC 实时观看 AI 在沙箱中的操作桌面。

## 2. 技术栈全景

| 层级/功能 | 技术选型 | 说明 |
|-----------|----------|------|
| Web 框架 | FastAPI + Uvicorn | 异步高性能，原生支持 ASGI |
| 数据校验 | Pydantic v2 | 全项目统一的模型层（领域模型 + API Schema）|
| 数据库 | PostgreSQL 14+ | 通过 `asyncpg` 驱动异步访问 |
| ORM | SQLAlchemy 2.0 | 全异步模式（`AsyncEngine`、`AsyncSession`）|
| 迁移工具 | Alembic | 数据库版本管理，启动时自动 `upgrade head` |
| 缓存/队列 | Redis 6+ | 异步客户端 `redis.asyncio`，使用 Redis Stream 实现消息队列 |
| LLM 接口 | OpenAI 兼容格式 | 通过 `AsyncOpenAI` 客户端调用，默认对接 DeepSeek |
| 浏览器自动化 | Playwright | 通过 CDP（Chrome DevTools Protocol）远程连接沙箱中的 Chrome |
| 沙箱环境 | Docker | 每个会话独立容器，内置 Ubuntu 22.04 + Chrome + VNC |
| 文件存储 | 阿里云 OSS / 腾讯云 COS | 用户上传的附件、AI 生成的结果文件 |
| 配置管理 | Pydantic Settings + YAML | `.env` 管理环境变量，`config.yaml` 管理业务配置 |
| 测试 | pytest | 基于 `TestClient` 的接口测试 |
| 包管理 | uv | 现代 Python 包管理器，替代 pip |

## 4. 架构哲学：Clean Architecture（整洁架构）

本项目是 Python 后端中**Clean Architecture** 的典范实现。核心思想是：

> **依赖关系只能由外向内，领域层（Domain）不依赖任何其他层。**

### 4.1 四层结构

```
┌──────────────────────────────────────────────────────────────┐
│  接口层（Interfaces）                                         │
│  职责：暴露 HTTP/WebSocket API，处理请求/响应序列化            │
│  依赖：应用层、基础设施层                                       │
├──────────────────────────────────────────────────────────────┤
│  应用层（Application）                                        │
│  职责：用例编排（Orchestration），协调领域对象完成业务流程      │
│  依赖：领域层、基础设施层                                       │
├──────────────────────────────────────────────────────────────┤
│  领域层（Domain）←←←←←←← 核心！                                │
│  职责：业务规则、领域模型、Agent 智能体逻辑、工具定义           │
│  依赖：无（只使用 Python 标准库和 Pydantic）                   │
├──────────────────────────────────────────────────────────────┤
│  基础设施层（Infrastructure）                                  │
│  职责：数据库访问、外部 HTTP 调用、Docker 管理、Redis 操作      │
│  依赖：领域层（实现领域层定义的 Protocol）                      │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 依赖反转（DIP）

领域层不直接引用基础设施层的任何类。相反，它只定义 **Protocol（协议）** 和 **Abstract Base Class（抽象基类）**：

- `LLM` 是一个 Protocol，`OpenAILLM` 是实现
- `Sandbox` 是一个 Protocol，`DockerSandbox` 是实现
- `IUnitOfWork` 是一个 ABC，`DBUnitOfWork` 是实现
- `SessionRepository` 是一个 ABC，`DBSessionRepository` 是实现

**依赖注入（DI）** 在 `service_dependencies.py` 中完成：手动将基础设施实现组装成应用服务，再注入到接口层（通过 FastAPI 的 `Depends`）。

### 4.3 为什么选择 Clean Architecture？

1. **可测试性**：领域层纯业务逻辑，不依赖数据库/HTTP，可单元测试
2. **可替换性**：今天用 DeepSeek，明天可无缝切换 Claude/OpenAI；今天用 Docker，明天可切换 Kata Containers
3. **可维护性**：业务逻辑集中在领域层，不会散落到路由、数据库查询、HTTP 调用中

## 5. 核心概念速览

在深入各层之前，先建立几个核心概念：

### 5.1 Session（会话）

一次完整的"聊天"就是一个 Session。它包含：
- 所有历史事件（events）
- 两个 Agent 的记忆（memories：planner + react）
- 关联的沙箱 ID（sandbox_id）
- 关联的任务 ID（task_id）
- 状态（pending → running → waiting/completed）

### 5.2 Event（事件）

前后端通信的基本单元。AI 的每一个"动作"都会产生一个事件：
- `MessageEvent`：AI 说的话
- `PlanEvent`：AI 生成的计划
- `StepEvent`：计划中的步骤开始/完成
- `ToolEvent`：工具被调用（calling → called）
- `DoneEvent`：任务结束

### 5.3 Task（任务）

当用户发送消息时，系统创建一个 **Task**。Task 包含：
- 输入流（input_stream）：接收用户消息
- 输出流（output_stream）：推送 AI 产生的事件
- 一个 **TaskRunner**（即 `AgentTaskRunner`）：真正执行 AI 逻辑

Task 基于 Redis Stream 实现，天然支持分布式部署。

### 5.4 Flow（流程）

`PlannerReActFlow` 是 AI 执行任务的"总指挥"。它是一个**状态机**，管理以下状态转移：

```
IDLE → PLANNING → EXECUTING → UPDATING → EXECUTING → ... → SUMMARIZING → COMPLETED → IDLE
```

### 5.5 Agent（智能体）

项目采用 **Multi-Agent** 架构：
- **PlannerAgent**：负责"思考"，将用户需求拆解为可执行的步骤列表（Plan）
- **ReActAgent**：负责"执行"，使用工具逐步完成每个步骤，并在最后总结

两者都继承 `BaseAgent`，共享 LLM 调用、工具调用、记忆管理等通用逻辑。

## 6. 配置体系

项目采用**双配置体系**：

### 6.1 `.env` —— 环境配置（敏感信息）

通过 `Pydantic Settings` 加载，支持环境变量覆盖：

```bash
SQLALCHEMY_DATABASE_URI=postgresql+asyncpg://...
REDIS_HOST=localhost
REDIS_PORT=6379
OSS_ACCESS_KEY_ID=...
COS_SECRET_ID=...
SANDBOX_ADDRESS=          # 留空则动态创建 Docker 容器
```

### 6.2 `config.yaml` —— 业务配置（非敏感）

包含 LLM 参数、Agent 行为参数、MCP 服务器列表、A2A 服务器列表：

```yaml
llm_config:
  base_url: https://api.deepseek.com/
  model_name: deepseek-reasoner
  temperature: 0.7
  max_tokens: 8192

agent_config:
  max_iterations: 100
  max_retries: 3

mcp_config:
  mcpServers:
    amap-maps-streamableHTTP:
      transport: streamable_http
      url: https://mcp.amap.com/mcp?key=xxx
```

**设计意图**：`config.yaml` 可由运营/管理员动态修改（如切换 LLM 模型、增删 MCP 服务），而 `.env` 由 DevOps 管理，两者解耦。

## 7. 启动流程（Lifespan）

`app/main.py` 中定义了 FastAPI 的 `lifespan`：

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    # 1. 重新初始化日志（uvicorn 会覆盖日志配置）
    setup_logging()
    
    # 2. 执行 Alembic 数据库迁移
    command.upgrade(alembic_cfg, "head")
    
    # 3. 初始化 Redis（5秒超时）
    await get_redis().init()
    
    # 4. 初始化 Postgres（10秒超时）
    await get_postgres().init()
    
    # 5. 初始化 OSS / COS
    await get_cos().init()
    
    yield  # ←←← 应用正式运行
    
    # 6. 关闭阶段：优雅关闭 Agent 服务 → Redis → Postgres → 存储
    await get_agent_service().shutdown()
    await get_redis().shutdown()
    ...
```

**关键细节**：
- Alembic 迁移完成后需要重新初始化日志，因为 alembic 会覆盖 logging 配置
- 关闭时先关闭 Agent 服务（等待正在运行的任务），再关闭基础设施
