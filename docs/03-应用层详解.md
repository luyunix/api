# Faber API —— 应用层（Application）详解

应用层是系统的"调度中心"。它本身不包含核心业务规则（那些属于领域层），而是负责：
- 接收接口层的请求，调用领域对象完成工作
- 协调多个领域对象之间的交互
- 管理事务边界（通过 UoW）
- 处理与具体用例相关的逻辑（如文件上传后的存储、沙箱状态检查）

---

## 1. 应用层服务一览

| 服务 | 文件 | 职责 |
|------|------|------|
| `AgentService` | `application/services/agent_service.py` | **核心服务**：管理 Agent 任务生命周期，处理 SSE 流式对话 |
| `SessionService` | `application/services/session_service.py` | 会话 CRUD、沙箱文件/Shell/VNC 操作 |
| `StatusService` | `application/services/status_service.py` | 系统健康检查（Postgres、Redis）|
| `FileService` | `application/services/file_service.py` | 文件上传下载（对接 OSS/COS）|
| `AppConfigService` | `application/services/app_config_service.py` | 读取 `config.yaml` 业务配置 |

## 2. AgentService：系统的核心引擎

`AgentService` 是**整个后端最复杂、最重要的服务**。它处理用户发来的每一条消息，驱动 AI Agent 执行任务，并管理任务的生命周期。

### 2.1 构造函数与依赖

```python
class AgentService:
    def __init__(
        self,
        uow_factory: Callable[[], IUnitOfWork],
        llm: LLM,
        agent_config: AgentConfig,
        mcp_config: MCPConfig,
        a2a_config: A2AConfig,
        sandbox_cls: Type[Sandbox],
        task_cls: Type[Task],
        json_parser: JSONParser,
        search_engine: SearchEngine,
        file_storage: FileStorage,
    ):
        self._uow_factory = uow_factory
        self._uow = uow_factory()          # 注意：这里创建了一个长期持有的 UoW
        self._llm = llm
        self._agent_config = agent_config
        self._mcp_config = mcp_config
        self._a2a_config = a2a_config
        self._sandbox_cls = sandbox_cls
        self._task_cls = task_cls
        self._json_parser = json_parser
        self._search_engine = search_engine
        self._file_storage = file_storage
```

**关键设计**：
- 通过 `uow_factory` 创建新的 UoW 实例，避免与其他服务共享数据库连接
- `sandbox_cls` 和 `task_cls` 是**类对象**（不是实例），实现延迟创建

### 2.2 chat()：处理用户消息的主流程

```python
async def chat(
    self,
    session_id: str,
    message: Optional[str] = None,
    attachments: Optional[List[str]] = None,
    latest_event_id: Optional[str] = None,
    timestamp: Optional[datetime] = None,
) -> AsyncGenerator[BaseEvent, None]:
```

这是 AgentService 最核心的方法，内部是一个复杂的**生产者-消费者模式**：

#### 阶段一：准备工作

```python
# 1. 检查会话是否存在
async with self._uow:
    session = await self._uow.session.get_by_id(session_id)
if not session:
    raise RuntimeError("任务会话不存在")

# 2. 获取或创建任务
task = await self._get_task(session)
```

`_get_task()` 逻辑：
- 如果 `session.task_id` 存在，从全局注册表获取已有 Task
- 如果不存在或已失效，返回 None

#### 阶段二：处理新消息（如果用户发送了消息）

```python
if message:
    # 会话不在运行中 → 需要创建新任务
    if session.status != SessionStatus.RUNNING or task is None:
        task = await self._create_task(session)
    
    # 更新会话的最后一条消息
    async with self._uow:
        await self._uow.session.update_latest_message(session_id, message, timestamp)
    
    # 查询附件的真实文件信息
    async with self._uow:
        db_attachments = [await self._uow.file.get_by_id(id) for id in attachments]
    
    # 创建用户消息事件
    message_event = MessageEvent(role="user", message=message, attachments=db_attachments)
    
    # 放入任务的输入队列
    event_id = await task.input_stream.put(message_event.model_dump_json())
    message_event.id = event_id
    yield message_event  # 先返回用户消息事件，前端可立即显示
    
    # 持久化到数据库
    async with self._uow:
        await self._uow.session.add_event(session_id, message_event)
    
    # 启动任务执行！
    await task.invoke()
```

#### 阶段三：消费输出事件（流式返回）

```python
while task and not task.done:
    # 从输出队列读取事件（非阻塞，block_ms=0）
    event_id, event_str = await task.output_stream.get(start_id=latest_event_id, block_ms=0)
    latest_event_id = event_id
    
    if event_str is None:
        continue  # 队列暂无新事件，继续轮询
    
    # 将 JSON 字符串反序列化为 Event 对象
    event = TypeAdapter(Event).validate_json(event_str)
    event.id = event_id
    
    # 重置未读消息数
    async with self._uow:
        await self._uow.session.update_unread_message_count(session_id, 0)
    
    yield event  # ← 流式推送给前端
    
    # 如果是终止事件，结束循环
    if isinstance(event, (DoneEvent, ErrorEvent, WaitEvent)):
        break
```

**轮询策略**：`block_ms=0` 表示非阻塞。如果队列空，立即返回 `None`，循环继续。这是为了避免阻塞 FastAPI 的协程，让其他请求有机会执行。

### 2.3 _create_task()：创建任务的完整流程

当需要启动新任务时：

```python
async def _create_task(self, session: Session) -> Task:
    # 1. 获取或创建沙箱
    sandbox = None
    if session.sandbox_id:
        sandbox = await self._sandbox_cls.get(session.sandbox_id)
    
    if not sandbox:
        sandbox = await self._sandbox_cls.create()
        session.sandbox_id = sandbox.id
        async with self._uow:
            await self._uow.session.save(session)
    
    # 2. 从沙箱获取浏览器实例
    browser = await sandbox.get_browser()
    
    # 3. 创建 AgentTaskRunner（领域对象）
    task_runner = AgentTaskRunner(
        uow_factory=self._uow_factory,
        llm=self._llm,
        agent_config=self._agent_config,
        mcp_config=self._mcp_config,
        a2a_config=self._a2a_config,
        session_id=session.id,
        file_storage=self._file_storage,
        json_parser=self._json_parser,
        browser=browser,
        search_engine=self._search_engine,
        sandbox=sandbox,
    )
    
    # 4. 创建 Task（基于 Redis Stream）
    task = self._task_cls.create(task_runner=task_runner)
    session.task_id = task.id
    async with self._uow:
        await self._uow.session.save(session)
    
    return task
```

**关键点**：
- 一个 Session 对应一个沙箱（长期复用）
- 一个 Session 的每次对话对应一个 Task（新的 Task 会复用已有沙箱）
- `AgentTaskRunner` 是领域层的对象，真正执行 AI 逻辑

### 2.4 SSE 断连问题的处理

这是生产环境中非常 tricky 的问题。当用户关闭浏览器或刷新页面时，SSE 连接断开，`sse_starlette` 会触发 **anyio cancel scope**，取消当前请求协程中的所有 `await`。

**问题**：如果在 `finally` 块中直接执行数据库操作，该操作也会被取消，导致 SQLAlchemy 连接池污染。

**解决方案**：

```python
finally:
    # 错误做法：await uow.session.update_unread_message_count(...) ← 会被取消！
    
    # 正确做法：创建独立的后台任务
    try:
        asyncio.create_task(self._safe_update_unread_count(session_id))
    except RuntimeError:
        pass  # 事件循环已关闭

async def _safe_update_unread_count(self, session_id: str) -> None:
    """在独立的 asyncio Task 中执行数据库操作，不受 cancel scope 影响"""
    try:
        uow = self._uow_factory()  # 全新的 UoW，全新的数据库连接
        async with uow:
            await uow.session.update_unread_message_count(session_id, 0)
    except Exception as e:
        logger.warning(f"后台更新未读消息计数失败: {e}")
```

这是一个**非常重要的工程经验**：在 SSE/WebSocket 等长连接场景中，`finally` 中的清理操作必须使用 `asyncio.create_task()` 脱离当前 cancel scope。

### 2.5 stop_session() 和 shutdown()

```python
async def stop_session(self, session_id: str) -> None:
    session = await self._uow.session.get_by_id(session_id)
    task = await self._get_task(session)
    if task:
        task.cancel()  # 取消后台执行的任务
    await self._uow.session.update_status(session_id, SessionStatus.COMPLETED)

async def shutdown(self) -> None:
    await self._task_cls.destroy()  # 销毁所有任务，释放沙箱资源
```

## 3. SessionService：会话管理

`SessionService` 处理与会话直接相关的操作，但不涉及 Agent 智能体逻辑。

### 3.1 基本 CRUD

```python
class SessionService:
    def __init__(self, uow_factory: Callable[[], IUnitOfWork], sandbox_cls: Type[Sandbox]):
        self._uow_factory = uow_factory
        self._uow = uow_factory()
        self._sandbox_cls = sandbox_cls

    async def create_session(self) -> Session:
        session = Session(title="新对话")
        async with self._uow:
            await self._uow.session.save(session)
        return session
    
    async def get_all_sessions(self) -> List[Session]:
        async with self._uow:
            return await self._uow.session.get_all()
```

### 3.2 沙箱操作代理

`SessionService` 充当了"沙箱操作代理"，所有对沙箱的读操作都通过它：

```python
async def read_file(self, session_id: str, filepath: str) -> FileReadResponse:
    # 1. 检查会话
    session = await self._uow.session.get_by_id(session_id)
    if not session:
        raise RuntimeError("会话不存在")
    
    # 2. 检查沙箱
    if not session.sandbox_id:
        raise NotFoundError("当前会话无沙箱环境")
    sandbox = await self._sandbox_cls.get(session.sandbox_id)
    if not sandbox:
        raise NotFoundError("沙箱不存在或已销毁")
    
    # 3. 调用沙箱读取文件
    result = await sandbox.read_file(filepath)
    if result.success:
        return FileReadResponse(**result.data)
    raise ServerRequestsError(result.message)
```

类似的方法还有 `read_shell_output()` 和 `get_vnc_url()`。

**设计意图**：路由层不应该直接操作沙箱，而是通过 SessionService 做权限检查（确保文件属于该会话的沙箱）。

## 4. StatusService：健康检查

```python
class StatusService:
    def __init__(self, checkers: List[HealthChecker]):
        self._checkers = checkers
    
    async def check_all(self) -> List[HealthStatus]:
        return [await checker.check() for checker in self._checkers]
```

使用了**策略模式**：`PostgresHealthChecker` 和 `RedisHealthChecker` 都实现 `HealthChecker` 协议，StatusService 不关心具体实现。

## 5. 应用层的设计原则

### 5.1 薄应用层

应用层不应该包含"如果用户说了 A 就做 B"这样的业务规则。这些属于领域层。应用层只负责：
- 获取输入 → 调用领域对象 → 返回输出
- 事务管理（`async with uow:`）
- 权限/状态校验

### 5.2 事务边界

```python
async with self._uow:
    await self._uow.session.save(session)
    # 退出上下文时自动 commit（无异常）或 rollback（有异常）
```

每个 `async with self._uow:` 块就是一个事务边界。应用层负责决定何时开启事务、何时提交。

### 5.3 异常转换

应用层捕获领域层抛出的原始异常，转换为应用层异常：

```python
try:
    sandbox = await self._sandbox_cls.get(session.sandbox_id)
except Exception:
    raise NotFoundError("沙箱不存在或已销毁")
```

这样接口层只需要处理应用层定义的标准异常，无需了解底层实现细节。
