# Faber API —— 接口层（Interfaces）详解

接口层是系统的"门面"，负责：
1. 暴露 HTTP API 和 WebSocket 端点
2. 请求/响应数据的序列化/反序列化
3. 全局异常处理
4. 依赖注入（将应用层服务注入到路由处理函数）

---

## 1. 路由结构

所有路由通过 `APIRouter` 模块化组织，最终在 `routes.py` 中聚合：

```python
# app/interfaces/endpoints/routes.py
from fastapi import APIRouter
from . import status_routes, app_config_routes, file_routes, session_routes

def create_api_routes() -> APIRouter:
    api_router = APIRouter()
    api_router.include_router(status_routes.router)
    api_router.include_router(app_config_routes.router)
    api_router.include_router(file_routes.router)
    api_router.include_router(session_routes.router)
    return api_router

router = create_api_routes()
```

然后在 `main.py` 中统一挂载：

```python
app.include_router(router, prefix="/api")
```

## 2. 依赖注入工厂（service_dependencies.py）

这是接口层**最核心的文件**，它负责将所有层的组件"组装"成可工作的服务。

### 2.1 设计模式：手动依赖注入

项目没有使用 `dependency-injector` 等 DI 框架，而是采用**纯函数 + FastAPI Depends** 的方式：

```python
# app/interfaces/service_dependencies.py
from fastapi import Depends
from sqlalchemy.ext.asyncio import AsyncSession

def get_agent_service(oss: Oss = Depends(get_oss)) -> AgentService:
    # 1. 读取业务配置
    app_config_repository = FileAppConfigRepository(config_path=settings.app_config_filepath)
    app_config = app_config_repository.load()
    
    # 2. 构建基础设施实现
    llm = OpenAILLM(app_config.llm_config)
    file_storage = OssFileStorage(bucket=settings.oss_bucket, oss=oss, uow_factory=get_uow)
    
    # 3. 组装应用服务
    return AgentService(
        uow_factory=get_uow,
        llm=llm,
        agent_config=app_config.agent_config,
        mcp_config=app_config.mcp_config,
        a2a_config=app_config.a2a_config,
        sandbox_cls=DockerSandbox,
        task_cls=RedisStreamTask,
        json_parser=RepairJSONParser(),
        search_engine=BingSearchEngine(),
        file_storage=file_storage,
    )
```

### 2.2 为什么是手动 DI？

1. **显式**：每个依赖的创建过程一目了然，没有"魔法"
2. **灵活**：某些依赖需要运行时参数（如 `app_config_filepath`），框架 DI 难以处理
3. **简单**：Python 的灵活语法让手动 DI 并不冗长

### 2.3 数据库会话的 per-request 生命周期

```python
async def get_db_session() -> AsyncSession:
    db = get_postgres()
    session_factory = db.session_factory
    async with session_factory() as session:
        try:
            yield session
        except Exception:
            await session.rollback()
            raise
```

FastAPI 的 `yield` 依赖确保：每个请求有独立的数据库会话，请求结束时自动提交或回滚。

## 3. SSE 流式响应（核心！）

`/sessions/{id}/chat` 是系统**最重要的接口**。它不使用普通的 JSON 响应，而是 **SSE（Server-Sent Events）**。

### 3.1 为什么用 SSE？

AI 执行任务不是瞬间完成的，可能需要：
- 生成计划（数秒）
- 多次搜索（每次数秒）
- 浏览器操作（点击、等待加载）
- 代码执行（可能数十秒）

如果用一个普通 POST 请求等待全部完成：
- HTTP 超时（通常 30-60 秒）
- 用户看不到进度，体验差

SSE 允许服务器**持续推送**多个事件，前端可以实时渲染 AI 的每一步动作。

### 3.2 实现细节

```python
from sse_starlette import EventSourceResponse, ServerSentEvent

@router.post("/{session_id}/chat")
async def chat(
    session_id: str,
    request: ChatRequest,
    agent_service: AgentService = Depends(get_agent_service),
) -> EventSourceResponse:
    
    async def event_generator():
        async for event in agent_service.chat(
            session_id=session_id,
            message=request.message,
            attachments=request.attachments,
            latest_event_id=request.event_id,
            timestamp=...,
        ):
            # 将领域事件转换为 SSE 事件
            sse_event = EventMapper.event_to_sse_event(event)
            yield ServerSentEvent(
                event=sse_event.event,      # 如 "message", "tool", "plan"
                data=sse_event.data.model_dump_json(),
            )
    
    return EventSourceResponse(event_generator())
```

### 3.3 EventMapper：领域事件 → SSE 事件

`EventMapper` 使用 **Python 反射**自动完成转换：

```python
class EventMapper:
    _cache_mapping: Optional[Dict[str, EventMapping]] = None
    
    @staticmethod
    def _get_event_type_mapping() -> Dict[str, EventMapping]:
        if EventMapper._cache_mapping is not None:
            return EventMapper._cache_mapping
        
        # 通过反射扫描 AgentSSEEvent Union 中的所有类
        sse_event_classes = get_args(AgentSSEEvent)
        mapping = {}
        
        for sse_event_class in sse_event_classes:
            # 提取 event 字段的 Literal 值
            event_field = sse_event_class.__annotations__["event"]
            event_type = event_field.__args__[0]
            
            mapping[event_type] = EventMapping(
                sse_event_class=sse_event_class,
                data_class=sse_event_class.__annotations__["data"],
                event_type=event_type,
            )
        
        EventMapper._cache_mapping = mapping
        return mapping
```

**设计优点**：
- 新增事件类型时，只需定义新的 `*SSEEvent` 类，无需修改映射逻辑
- 类级缓存避免每次反射的性能损耗

## 4. WebSocket VNC 代理

`/sessions/{id}/vnc` 是一个 WebSocket 端点，让用户能在浏览器中"看到"沙箱桌面：

```python
@router.websocket("/{session_id}/vnc")
async def vnc_websocket(websocket: WebSocket, session_id: str):
    # 1. 协商子协议（noVNC 首选 binary）
    protocols = websocket.headers.get("sec-websocket-protocol", "").split(",")
    selected = "binary" if "binary" in protocols else "base64"
    await websocket.accept(subprotocol=selected)
    
    # 2. 获取沙箱的 VNC WebSocket URL
    sandbox_vnc_url = await session_service.get_vnc_url(session_id)
    
    # 3. 建立到沙箱 VNC 的连接
    async with websockets.connect(sandbox_vnc_url) as sandbox_ws:
        # 4. 创建双向转发协程
        async def forward_to_sandbox():
            while True:
                data = await websocket.receive_bytes()
                await sandbox_ws.send(data)
        
        async def forward_from_sandbox():
            while True:
                data = await sandbox_ws.recv()
                await websocket.send_bytes(data)
        
        # 5. 并行运行，任一方向断开则关闭全部
        task1 = asyncio.create_task(forward_to_sandbox())
        task2 = asyncio.create_task(forward_from_sandbox())
        done, pending = await asyncio.wait(
            [task1, task2], return_when=asyncio.FIRST_COMPLETED
        )
        for task in pending:
            task.cancel()
```

### 4.1 架构意义

这是**透明代理模式**：Faber 后端本身不处理 VNC 协议，只是将客户端（noVNC）和沙箱（x11vnc/websockify）之间的数据做双向转发。这样：
- 后端无状态，不占用 VNC 解析资源
- 天然支持认证/鉴权（在 `get_vnc_url` 阶段校验 session）
- 沙箱和客户端无需直接通信

## 5. 通用响应封装

所有 API 使用统一的响应结构：

```python
class Response(BaseModel, Generic[T]):
    code: int = 200
    msg: str = "success"
    data: Optional[T] = Field(default_factory=dict)
    
    @staticmethod
    def success(data=None, msg="success") -> "Response[T]":
        return Response(code=200, msg=msg, data=data)
    
    @staticmethod
    def fail(code, msg, data=None) -> "Response[T]":
        return Response(code=code, msg=msg, data=data)
```

使用泛型 `Response[CreateSessionResponse]`，FastAPI 可以自动生成精确的 OpenAPI Schema。

## 6. 全局异常处理

`exception_handlers.py` 注册了三个异常处理器：

```python
def register_exception_handlers(app: FastAPI):
    @app.exception_handler(AppException)
    async def app_exception_handler(req, e):
        return JSONResponse(status_code=e.status_code, content=Response.fail(...))
    
    @app.exception_handler(HTTPException)
    async def http_exception_handler(req, e):
        return JSONResponse(status_code=e.status_code, content=Response.fail(...))
    
    @app.exception_handler(Exception)
    async def exception_handler(req, e):
        logger.exception(f"Unhandled Exception: {e}")
        return JSONResponse(status_code=500, content=Response.fail(
            500, f"服务器异常: {e}", 
            data={"traceback": traceback.format_exc()} if app.debug else {}
        ))
```

**分层异常设计**：
- `AppException`：业务异常（如会话不存在、沙箱已销毁），返回 4xx
- `HTTPException`：FastAPI 标准异常（如参数校验失败），透传状态码
- `Exception`：兜底，500 错误，开发模式返回堆栈
